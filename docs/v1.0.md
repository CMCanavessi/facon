## Overview

Facón 1.0 is a classical handcrafted chess engine. It uses no neural networks or external data — all knowledge of chess is encoded explicitly in the evaluation function and search heuristics.

The engine is structured as a pipeline:

```
UCI input → position setup → move generation → search → evaluation → UCI output
```

All components are single-threaded. The engine uses one board instance, one transposition table, and one time manager as global state.

---

## Board Representation

The board state is held in the `Board` struct and uses two redundant representations kept in sync at all times:

### Bitboards

A `Bitboard` is a `uint64_t` where each bit corresponds to one square. Bit 0 = a1, bit 1 = b1, ..., bit 63 = h8.

```
Bit layout:
  a8 b8 c8 d8 e8 f8 g8 h8   ← bits 56..63
  a7 b7 c7 d7 e7 f7 g7 h7   ← bits 48..55
  ...
  a1 b1 c1 d1 e1 f1 g1 h1   ← bits 0..7
```

The engine maintains:

- `pieces[7]` — one bitboard per piece type (indexed by `PieceType`). `pieces[PAWN]` contains all pawns of both colors.
- `by_color[2]` — one bitboard per color. `by_color[WHITE]` contains all white pieces.

To get all white pawns: `pieces[PAWN] & by_color[WHITE]`.

Bitboards are fast for bulk operations: attack generation, occupancy checks, population counts.

### Piece Array

`piece_on[64]` maps each square to the `Piece` on it (`NO_PIECE` if empty). Fast for "what is on square X?" queries, which are common in move generation and evaluation.

### History Stack

Before each move, the parts of the state that cannot be recovered by reversing the move are saved to a `StateInfo` entry on the history stack:

- Captured piece
- En passant square
- Castling rights
- Half-move clock
- Zobrist hash

`make_move` pushes to this stack; `unmake_move` pops from it and restores.

## Magic Bitboards

Sliding piece attacks (rooks, bishops, queens) depend on the current occupancy of the board. Computing them naively would be too slow. Facón uses **magic bitboards** — a lookup table technique that computes sliding attacks in O(1) per piece.

### Principle

For a given square and occupancy:

1. Mask the relevant squares (the attack ray, excluding the edges).
2. Multiply the masked occupancy by a precomputed "magic number".
3. Shift right to get a small index.
4. Look up the attack bitboard in a precomputed table.

```cpp
Bitboard rook_attack(Square s, Bitboard occ) {
    occ &= magic_rook[s].mask;
    occ *= magic_rook[s].magic;
    occ >>= magic_rook[s].shift;
    return rook_table[s][occ];
}
```

### Magic Number Generation

Facón generates its own magic numbers at startup using a randomized trial-and-error search with a fixed seed (`0xFA60E8CE55E0A000`). This guarantees:

- No reliance on hardcoded magic numbers borrowed from other engines.
- Reproducible results across runs.
- Correct tables verified implicitly by the perft test suite.

The search uses **sparse random numbers** (AND of two random values) which empirically find valid magics faster.

### Occupancy Enumeration

During initialization, all relevant occupancy subsets for each square are enumerated using the **carry-rippler trick**:

```cpp
Bitboard subset = 0;
do {
    // process subset
    subset = (subset - mask) & mask;
} while (subset);
```

This iterates over all 2^N subsets of a given mask in O(2^N) time.

### Between and Line Tables

Two auxiliary tables are also computed during initialization:

- `between_bb[s1][s2]` — squares strictly between two squares on the same ray
- `line_bb[s1][s2]` — the full ray through both squares (used for pin detection)

## Zobrist Hashing

Each position is identified by a 64-bit Zobrist hash, computed as the XOR of random values for each feature of the position:

- Each piece on each square: `piece_square[piece][square]`
- Castling rights: `castling[rights]` (4-bit mask, 16 entries)
- En passant file: `en_passant[file]` (only when an EP capture is possible)
- Side to move: `side_to_move` (XORed in when it is Black's turn)

All random values are generated once at startup using `std::mt19937_64` with a fixed seed.

The hash is updated **incrementally**: `make_move` and `unmake_move` XOR only the values that change, rather than recomputing the full hash. This makes hashing essentially free.

The hash is used by the transposition table and by the repetition detection algorithm.

## Move Encoding

A move is a `uint32_t` with the following layout:

```
Bits  0.. 5  from square      (0..63)
Bits  6..11  to square        (0..63)
Bits 12..13  move type        (NORMAL=0, PROMOTION=1, EN_PASSANT=2, CASTLING=3)
Bits 14..15  promotion piece  (0=Knight, 1=Bishop, 2=Rook, 3=Queen)
```

The sentinel value `MOVE_NONE = 0` is a1→a1 NORMAL, which is never a legal move.

Promotions always generate all four promotion pieces. The search will naturally prefer queen promotions, but underpromotions are available when they are the best move.

## Move Generation

Move generation is pseudo-legal: generated moves respect piece movement rules but may leave the king in check. Legality is checked by `board.is_legal(m)` before each move is made in the search.

`is_legal` works by making the move on a copy of the board and verifying that the king is not attacked afterward.

### Generation Order

Moves are generated piece by piece in this order: pawns, knights, bishops, rooks, queens, king. Two entry points are provided:

- `generate_all_moves` — all pseudo-legal moves (used in main search)
- `generate_captures` — captures and promotion captures only (used in quiescence search)

### Pawn Moves

Pawns are the most complex piece:

- **Single push**: one square forward into an empty square.
- **Double push**: from the starting rank, if both intermediate squares are empty. Generated as a shift of the single-push result, so the intermediate square is guaranteed empty.
- **Captures**: diagonal, intersected with enemy piece bitboard.
- **En passant**: detected using the stored `ep_square`. The capturing pawn is found by applying the opponent's pawn attack pattern from `ep_square`.
- **Promotions**: pawns on the 7th rank (White) or 2nd rank (Black) generate promotion moves for all four pieces.

### Castling

Castling conditions checked at generation time:
1. The relevant castling right is still set.
2. All squares between king and rook are empty.
3. The king is not in check.
4. The king does not pass through or land on an attacked square.

The king's current square is not checked for attack here — that is already guaranteed by the in-check check.

## Search

### Iterative Deepening

The top-level search (`Search::go`) runs negamax at increasing depths (1, 2, 3, ...) until time runs out. Each iteration starts fresh but benefits from the transposition table populated by previous iterations, which provides good move ordering and frequent TT hits.

If time expires mid-iteration, the result is discarded and the last fully completed iteration is used.

### Negamax Alpha-Beta

The core search is negamax with alpha-beta pruning. Negamax exploits the zero-sum property of chess: the score from the opponent's perspective is the negation of our score. This allows a single recursive function to handle both sides.

Alpha-beta pruning cuts branches that cannot affect the final result:
- **Beta cutoff** (fail-high): the current position is too good — the opponent will avoid it.
- **Fail-low**: no move improved alpha — we store an upper bound.
- **Exact score**: alpha was improved but beta was not exceeded.

### Draw Detection

Before generating moves, the search checks for draws:
- **Repetition**: `board.is_repetition()` checks the Zobrist hash against the history stack, stepping back two plies at a time (same side to move) up to the last irreversible move.
- **50-move rule**: `half_move_clock >= 100`.

Both checks are skipped at the root (`ply == 0`) to ensure the engine always returns a real move.

### Checkmate and Stalemate

Detected after the move loop when no legal moves were found:
- **Checkmate**: `board.in_check()` is true → return `-SCORE_MATE + ply`. The `+ ply` adjustment ensures the engine prefers faster mates (lower ply = sooner = higher score).
- **Stalemate**: not in check → return `SCORE_DRAW` (0).

### Quiescence Search

Called at `depth == 0` instead of returning `evaluate()` directly. Searches only captures until the position is "quiet", preventing the horizon effect — the engine thinking it gained a piece without seeing the immediate recapture.

**Stand pat**: before searching captures, the static evaluation is used as a lower bound. If it already exceeds beta, we prune immediately (assumption: the side to move can always find a non-capture move that maintains the evaluation).

### Move Ordering

Good move ordering dramatically improves alpha-beta efficiency. Facón 1.0 uses three tiers:

| Tier | Score | Description |
|------|-------|-------------|
| TT move | 1,000,000 | Best move from a previous search at this position |
| Captures | 100,000 + MVV-LVA | Most Valuable Victim / Least Valuable Attacker |
| Quiet moves | 0 | No ordering in 1.0 |

**MVV-LVA** (Most Valuable Victim, Least Valuable Attacker): captures are ordered by victim value descending and attacker value ascending. A pawn capturing a queen scores higher than a queen capturing a pawn.

Sorting is done with selection sort, which is simple and fast enough for move lists of 30–50 moves.

## Evaluation

The evaluation function returns a score in centipawns (100 = one pawn advantage) from the perspective of the side to move. Positive = good for the mover.

### Material Values

| Piece | Value |
|-------|-------|
| Pawn | 100 |
| Knight | 320 |
| Bishop | 330 |
| Rook | 500 |
| Queen | 900 |
| King | 0 (no material value) |

The bishop is worth slightly more than the knight, reflecting the general preference for bishops in open positions.

### Piece-Square Tables

Each piece type has a table of 64 bonuses/penalties, one per square. Tables are defined from White's perspective (a1=index 0, h8=index 63). For Black, the square index is mirrored vertically: `idx = sq ^ 56`.

Key design decisions:

- **Pawns**: reward center control and advancement. Large bonus on the 7th rank.
- **Knights**: strongly reward center squares (d4, e4, d5, e5), penalize edges and corners.
- **Bishops**: reward long diagonals and center activity.
- **Rooks**: reward the 7th rank and slightly penalize rim files.
- **Queens**: discourage early development, reward center activity.
- **King (middlegame)**: reward castled positions (g1/c1 and equivalents), strongly penalize the center.
- **King (endgame)**: reward centralization — the king becomes an active piece.

### Game Phase Interpolation

The king uses two separate tables (middlegame and endgame) interpolated by a phase value computed from remaining material:

```
Phase contributions: Knight=1, Bishop=1, Rook=2, Queen=4
Maximum phase = 24 (starting position minus pawns and kings)
```

The phase value ranges from 0 (full endgame) to 256 (full middlegame). The interpolation formula:

```cpp
score = (mg * phase + eg * (256 - phase)) / 256;
```

This prevents the king from abruptly changing behavior when the last piece is traded.

## Transposition Table

The transposition table (TT) caches search results to avoid re-analyzing positions that have been reached by a different move order.

### Entry Layout

Each entry is exactly 16 bytes (no padding):

| Field | Type | Size | Description |
|-------|------|------|-------------|
| `hash` | `uint64_t` | 8 bytes | Full Zobrist hash for collision verification |
| `move` | `uint32_t` | 4 bytes | Best move found |
| `score` | `int16_t` | 2 bytes | Search score (mate-distance adjusted) |
| `depth` | `int8_t` | 1 byte | Search depth |
| `bound` | `uint8_t` | 1 byte | Bound type (EXACT / LOWER / UPPER) |

### Indexing

The table size is always a power of two. Indexing uses a bitmask: `hash & (size - 1)`. This is faster than modulo and is correct because the hash is uniformly distributed.

### Replacement Strategy

Always-replace: new entries unconditionally overwrite old ones. Exception: if the new entry has no best move (`MOVE_NONE`) but the existing entry has one for the same position, the old move is preserved. This prevents losing the PV move when a shallower re-search overwrites a deeper result.

### Bound Types

- **EXACT**: the score is the true minimax value. All moves were searched and alpha was improved.
- **LOWER**: a beta cutoff occurred. The true score may be higher.
- **UPPER**: all moves failed low. The true score may be lower.

### Mate Score Adjustment

Mate scores encode distance-to-mate in plies. Before storing in the TT, the score is adjusted to be relative to the current ply (`score + ply` for positive mates). On retrieval, this is reversed (`score - ply`). This ensures a TT hit from a different search depth produces the correct mate distance at the root.

## Time Management

The time manager (`TimeManager`) decides how many milliseconds to spend on each move.

### Modes

- **Normal clock** (`wtime`/`btime` + optional `winc`/`binc`): divides remaining time by an assumed 30 moves remaining, adds 75% of the increment. Result is capped at 50% of remaining time to prevent time trouble.
- **Fixed movetime** (`movetime X`): spends exactly X milliseconds (minus a small safety margin).
- **Fixed depth** (`depth X`): searches to exactly depth X regardless of time.
- **Infinite** (`go infinite`): searches indefinitely until a `stop` command is received.

### Stop Signal

The `stop` field is a `std::atomic<bool>` written by the UCI thread and read by the search thread. The search checks `TM.should_stop()` every 2048 nodes to minimize overhead while remaining responsive.

## UCI Protocol

Facón implements the full UCI command set required for GUI compatibility.

### Supported Commands

| Command | Description |
|---------|-------------|
| `uci` | Print engine identity and options |
| `isready` | Synchronization — always responds `readyok` immediately |
| `ucinewgame` | Clear the TT and reset the board |
| `position startpos [moves ...]` | Set up a position from the start |
| `position fen <fen> [moves ...]` | Set up a position from FEN |
| `go [options]` | Start searching |
| `stop` | Abort the current search |
| `setoption name Hash value <mb>` | Set the TT size |
| `quit` | Exit the engine |
| `d` | Display the current board (debug, not part of UCI spec) |

### Info Output

During search, the engine outputs one `info` line per completed iteration:

```
info depth <d> score cp <s> nodes <n> nps <n> time <ms> hashfull <pm> pv <move>
```

Mate scores are reported as `score mate <n>` where n is moves to mate (positive = we are mating, negative = we are being mated).

## Known Limitations

These are known weaknesses in 1.0 that will be addressed in future versions:

- **No null move pruning**: the search explores every branch without skipping turns, which is significantly slower than engines that use this technique.
- **No killer moves or history heuristic**: quiet move ordering is random. This is a major source of inefficiency.
- **No late move reductions (LMR)**: all moves are searched at full depth.
- **No SEE (Static Exchange Evaluation)**: the quiescence search cannot distinguish winning captures from losing ones. The engine will sometimes initiate losing exchanges it cannot foresee.
- **Basic evaluation**: no king safety beyond the PST, no pawn structure analysis, no mobility, no open file detection.
- **Always-replace TT**: a depth-based replacement strategy would retain more useful entries under pressure.
- **Single-threaded**: no support for Lazy SMP or any other form of parallelism.
- **No ponder**: the engine does not think on the opponent's time.
