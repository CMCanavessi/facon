## Overview

Facon 1.4 "Hoja" is the largest single-version improvement in the project's history: +430 Elo over 1.3 (Ordo ~2330 vs ~1900). The gains come from three compounding sources:

1. **Speed** (+134% NPS from make/unmake, move score precomputation, LMR table): the engine searches ~2x faster, reaching 2-3 plies deeper at the same time control.

2. **Pruning** (SEE, reverse futility, move-level futility, check extension): the engine skips unproductive branches and extends important ones. Bench nodes at depth 12 dropped from 19.1M (pre-1.4) to 7.9M (-59%).

3. **Evaluation** (mobility, open files, rook on 7th, bishop pair, knight outposts): the engine makes better decisions at every node. Combined with deeper search, the effect is multiplicative.

Final NPS (release build, bench depth 15):

| Platform | 1.3 NPS | 1.4 NPS | Improvement |
|----------|---------|---------|-------------|
| Linux (VM Debian, Ryzen 7 1700) | ~1,344K | ~2,061K | +53% |
| Windows 10 (native, Ryzen 7 1700) | ~1,142K | ~2,188K | +92% |

Note: Windows 1.4 is faster than Linux 1.4 despite Linux traditionally being faster. The MSVC native compiler produces better optimized code than MinGW cross-compilation for the make/unmake hot path. This is the first version where Windows outperforms Linux.

Unchanged from 1.3: board representation, magic bitboards, move generation (except quiet queen promotions in generate_captures), NMP, triangular PV array, PV repetition detection, king safety, killer heuristic, history heuristic, LMR (formula unchanged, only precalculated), aspiration windows (formula unchanged, added fail-high extension), UCI threading, UCI option handling, heartbeat, currmove output, pawn structure evaluation.

## Search Speedups

### Make/unmake legality (search.cpp)

The single largest speedup. In 1.3, every pseudo-legal move was tested with `is_legal()`, which copies the entire Board struct (~700 bytes), makes the move on the copy, checks if the king is in check, and discards the copy. With ~30 pseudo-legal moves per node and ~1.4M nodes/second, this was ~30GB/s of memcpy saturating L1/L2 cache.

In 1.4, the move is made on the real board. If the king is left in check (illegal move), it is unmade and skipped. Legal moves proceed to recursive search and are unmade after. The check is: `board.is_attacked(board.king_square(~board.side_to_move), board.side_to_move)` -- after make_move, side_to_move has flipped, so "our" king is `~side_to_move` and the potential attacker is `side_to_move`.

Impact: +130% NPS. The improvement is multiplicative because eliminating the memcpy frees cache bandwidth for everything else (TT probes, history lookups, PST access).

`is_legal()` is unchanged and still used by non-hot-path callers: perft, move_to_san, PV walk, root move counting, parse_move.

### Move scores computed once (search.cpp)

In 1.3, `sort_moves()` used selection sort where `move_score()` was called on every comparison -- O(n^2) calls for n moves. In 1.4, scores are computed once into a parallel `int scores[MAX_MOVES]` array (O(n) calls), and the selection sort swaps both the move and its score in parallel.

Impact: +9% NPS.

### LMR table precalculated (search.h, search.cpp, main.cpp)

The per-move `std::log(depth) * std::log(move) / LMR_DIVISOR` in the negamax hot path is replaced with a lookup into `LMR_table[MAX_PLY][MAX_MOVES]`, initialized once at startup by `init_lmr_table()`. Same formula, same results.

Impact: ~0% NPS (the log calls were already fast relative to the rest of negamax).

### Quiet queen promotions in qsearch (movegen.cpp, movegen.h)

`generate_captures()` (used by quiescence search) now includes quiet queen promotions: a pawn pushing to the 8th rank without capturing. Gaining a queen for free is as significant as a capture and was previously invisible to qsearch. Only queen promotion is generated; underpromotions would add noise without benefit.

Impact: +0.39% bench nodes (endgame positions with passed pawns most affected).

## Check Extension

When the side to move is in check, the search depth is extended by 1 ply (`if (in_check) depth++`). Being in check is a forcing situation with very few legal responses (often 1-3). Without the extension, check sequences at the search horizon would drop into quiescence, where only captures are considered -- missing critical evasions.

The extension is applied BEFORE the `depth == 0` quiescence entry check. A node arriving at depth 0 while in check gets extended to depth 1 (full search with all moves) instead of entering qsearch.

Impact: +30 Elo (self-play, 1000 games).

## Static Exchange Evaluation (SEE)

### Concept

SEE simulates the full exchange sequence on a target square and returns the material gain or loss from the perspective of the capturing side. Example: if a queen captures a defended pawn (QxP), SEE returns -800 (gain pawn +100, lose queen -900).

### Implementation (search.cpp)

The `see(board, m)` function uses a gain array and negamax-style propagation:

1. `gain[0]` = value of the initial capture (or promotion gain for promotion moves).
2. Remove the moving piece from a local occupancy bitboard.
3. Find all attackers to the target square using the updated occupancy (`all_attackers_to(sq, occ)` -- this discovers x-ray attackers behind the removed piece).
4. Find the cheapest attacker of the defending side (PAWN < KNIGHT < BISHOP < ROOK < QUEEN < KING).
5. `gain[d] = value_of_piece_just_captured - gain[d-1]` (negamax).
6. Remove that attacker from occupancy, recompute attackers, switch sides. Repeat until no attackers remain.
7. Propagate backwards: at each step, the side can choose NOT to recapture if it would make them worse off.

### all_attackers_to (board.h, board.cpp)

New function that returns a bitboard of ALL pieces (both colors) attacking a square, using a given occupancy for slider computation. Unlike the existing `attackers_to(sq, color)`, this takes an occupancy parameter -- essential for SEE because as pieces are removed from the exchange, sliders behind them become visible (x-ray discovery).

### Usage

In quiescence search, captures with `see(board, m) < 0` are skipped. Shortcut: if the victim is worth >= the attacker (`PIECE_VALUE[victim_type] >= PIECE_VALUE[attacker_type]`), the capture is always winning or equal and SEE is not called.

En passant is handled: `gain[0] = PAWN_VALUE`, and the captured pawn is removed from occupancy. Promotions adjust `gain[0]` for the promotion piece value minus pawn value.

## Futility Pruning

Two levels of futility pruning, both gated behind `can_prune` (not in check, not at root, depth <= max(RFP_MAX_DEPTH, FUTILITY_MAX_DEPTH)):

### Reverse Futility Pruning (RFP) -- node level

At depth 1-3 (`RFP_MAX_DEPTH = 3`), if the static evaluation minus a margin already exceeds beta, the node is pruned entirely:

```
if (static_eval - RFP_MARGIN * depth >= beta) return static_eval;
```

`RFP_MARGIN = 100` centipawns per depth level: d1=100, d2=200, d3=300.

This is like a cheaper version of NMP: if the position is so good that even losing 100-300cp per ply would still be above beta, there is no need to search.

### Move-level futility -- move level

At depth 1-2 (`FUTILITY_MAX_DEPTH = 2`), quiet moves (non-capture, non-promotion) are skipped if:

```
if (static_eval + FUTILITY_MARGIN * depth <= alpha) { unmake; continue; }
```

`FUTILITY_MARGIN = 150` centipawns per depth level: d1=150, d2=300.

At least one legal move is always searched (the check `legal_count > 1` ensures checkmate and stalemate are still detected).

### static_eval optimization

`evaluate()` is only called when `can_prune` is true (depth <= 3, not in check, not root). At depth 4+ neither RFP nor move-level futility fires, so the evaluate call would be wasted work. This avoids calling the now-heavier evaluation function (with mobility, outpost checks, etc.) at every node.

## Evaluation -- Positional Terms

All new terms are computed in `positional_eval()` (eval.cpp), called once per `evaluate()`. Returns a score from White's perspective. The function iterates over both colors, computing all terms symmetrically.

### Mobility

Pseudo-legal squares per piece, excluding own pieces and squares attacked by enemy pawns (unsafe squares). The mobility area `mob_area = ~our_pieces & ~enemy_pawn_attacks` ensures pieces are not rewarded for "mobility" into squares they would immediately be captured on.

| Piece | Bonus per square | Typical range | Rationale |
|-------|-----------------|---------------|-----------|
| Knight | 4cp | 4-8 squares | Moderate -- knights have fixed mobility |
| Bishop | 5cp | 5-13 squares | Highest -- bishops benefit most from open diagonals |
| Rook | 2cp | 5-14 squares | Low per-square -- rooks have high base count even when passive |
| Queen | 1cp | 9-27 squares | Minimal -- avoids overvaluing queens for their inherently high mobility |

### Open and semi-open files

For each rook, check its file for friendly and enemy pawns:
- No pawns of either color on the file: `ROOK_OPEN_FILE = +20cp`
- No friendly pawns but enemy pawns present: `ROOK_SEMI_OPEN_FILE = +10cp`

### Rook on 7th rank

`ROOK_ON_7TH = +20cp` when a rook is on the opponent's 2nd rank (relative rank 6, 0-indexed). Strong because it attacks pawns on their starting rank and confines the enemy king to the back rank.

### Bishop pair

`BISHOP_PAIR_BONUS = +30cp` when both bishops are present (`popcount(bishops) >= 2`). Two bishops complement each other's color coverage and are stronger together than the sum of their parts.

### Knight outposts

`KNIGHT_OUTPOST = +20cp` for a knight on ranks 4-6 (relative) where no enemy pawn on adjacent files can advance to attack the square. The check uses a forward mask from the knight's perspective: only enemy pawns ahead of the knight (from the enemy's viewpoint) can threaten it.

## Transposition Table -- Depth-Preferred Replacement

### Problem

In 1.3, `store()` always overwrites the existing entry. A shallow search at depth 2 can evict a deep result at depth 15 for a different position. Deep results are expensive to recompute and produce more cutoffs.

### Solution

`store()` now checks before writing. It only overwrites if:
1. The slot is empty (`BOUND_NONE`).
2. Same position (hash match) -- newer data for the same position is always more current.
3. New depth >= stored depth -- deeper or equal search is more valuable.

If none of these hold (different position, shallower depth), the store is refused.

### TTEntry::depth type fix

Changed from `int8_t` (max 127) to `uint8_t` (max 255). `MAX_PLY` is 128, which overflows `int8_t` to -128 -- corrupting the depth comparison in `probe()`. Entry size unchanged (16 bytes, same struct layout).

## Time Management

### MOVES_TO_GO: 30 to 25

More time per move (~20% increase). Realistic for games averaging 35-45 moves at this strength level. The increment compensates for the last few moves.

### Progressive easy-move

Replaces the one-shot x0.40 reduction from 1.3. After 5 stable iterations (same PV move, score change < 5cp) at depth >= 10, each subsequent stable iteration multiplies the soft limit by `EASY_REDUCE_FACTOR = 0.95`. The effect accumulates: 0.95^5 = 0.77, 0.95^10 = 0.60.

Tracked via `easy_cumulative_` (initialized to 1.0 each search). On PV change or score drop >= 30cp, the cumulative reduction is reversed via `cancel_easy_move(easy_cumulative_)` and the counter resets. Score changes of 5-29cp pause the counter (stop reducing) but do not reverse accumulated reductions.

### AW fail-high time extension

New. After the aspiration window loop resolves, if there were fail-highs, time is extended proportionally: `1.0 + min(aw_fail_highs * 0.10, 0.50) * depth_scale`. Capped at x1.50.

Fail-high means the engine found something better than expected. More time allows the window to resolve correctly and find the real best move.

### EXTENSION_FULL_DEPTH: 18 to 15

The quadratic depth scaling factor `(depth/EXTENSION_FULL_DEPTH)^2` now reaches 1.0 at depth 15 instead of 18. At depth 10, the factor is 0.44 (was 0.31) -- PV changes in the middlegame get more time. Balanced by the progressive easy-move which returns time when the position stabilizes.

### Time cap: remaining/3 to remaining*2/5

The hard cap on base_time and hard_limit is relaxed from 33% to 40% of remaining clock. More permissive in endgames where the tree is narrow and deeper search is valuable.

### Mate reduction guard

`reduce_time("mate found")` now requires `last_score > 0` (we are winning). Previously it fired on any mate score including being mated (`last_score < 0`), which reduced time exactly when the engine needed MORE time to find a defense.

### soft_stop reorder

The first `soft_stop()` check (immediately after the info line) was removed. TM extensions (PV change, score drop, AW fail-high) now always run before checking whether to stop. Previously, a PV change at the end of an iteration could be missed if the soft limit had already expired.

### go depth X fix

`go depth N` without a clock previously activated the TM fallback (600ms soft / 1000ms hard), causing spurious time extensions and premature stops. Now detected in `start()`: if `depth_limit > 0` and no clock, sets `infinite = true`.

### movestogo

The `movestogo` parameter from `go ... movestogo N` is now parsed and used instead of the hardcoded `MOVES_TO_GO` constant. Cleared in `reset()`.

## Infrastructure

### bench command (uci.cpp, uci.h)

`bench [depth] [verbose]` searches 10 hand-crafted positions and reports per-position nodes, time, and total NPS. Default depth: 15. Options can be in any order (`bench 10 verbose`, `bench verbose 10`).

In quiet mode (default), all UCI output (info lines, currmove, TM messages, ST debug) is suppressed via `NullBuffer` redirecting `std::cout`. Only the position summaries and final result are printed. `bench verbose` shows full output.

Bench positions (10, all original) cover: opening/TT, LMR+history, qsearch/tactical, NMP, closed/LMR, king safety, pawn structure, rook endgame, zugzwang/NMP guards, mopup K+R vs K.

### Forced move instant-play (search.cpp)

Legal moves are counted before the iteration loop. If exactly one legal move exists, the engine plays it immediately without searching. A minimal `info depth 0 score cp 0 nodes 0 time 0 pv <move>` line is emitted for GUI/match runner compatibility (fastchess crashes without at least one info line before bestmove).

Root move count is computed once and reused for all iterations instead of recounting per iteration.

### FACON_DEBUG build mode (CMakeLists.txt, search.h, search.cpp)

`cmake -DFACON_DEBUG=ON` enables diagnostic counters in SearchStats: `lmr_attempted`, `lmr_re_searched`, `nmp_attempted`, `nmp_cutoffs`, `tt_cutoffs`. Reported via `info string ST:` after each completed iteration. All behind `#ifdef FACON_DEBUG` -- zero cost in release builds.

### Dev codename detection (CMakeLists.txt)

Changed from `STREQUAL "dev"` to `MATCHES "^dev"` so codenames like `dev-rc1`, `dev-rc2` are handled automatically. The version string and binary name include the full codename: `facon-1.4-dev-rc1`.

## Known Limitations

- **No endgame recognition**: K+R vs K+B and similar endings where the material advantage is below `MOPUP_THRESHOLD` (300cp) have no positional direction. The engine holds the advantage but may not make progress. Planned for 1.6 (material signature dispatch).
- **No pawn shelter/storm evaluation**: the engine has no explicit bonus for castled king safety beyond PSTs. May delay castling in some positions. Planned for 1.5.
- **No TT aging**: `hashfull` reports all occupied slots regardless of relevance. Entries from abandoned branches persist until overwritten. Planned for 1.5.
- **Single-threaded**: no Lazy SMP. Planned for 1.8.
- **No Syzygy tablebases**: endgame play relies entirely on search depth + mopup. Planned for 1.5.
- **Bench positions unbalanced**: position 3 takes ~37% of total bench time while position 9 takes ~0.03%. Per-position NPS analysis is unreliable for the extremes. Rebalancing planned.
