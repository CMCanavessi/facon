## Overview

Facon 1.5 "Espiga" gains approximately +220 Elo over 1.4 "Hoja" (~2550 vs ~2330 Ordo). Unlike 1.4, where the gain came from a balanced mix of speed (+134% NPS) and new search/eval features, 1.5's improvement is dominated by a single fix: the `game_phase()` function had been returning the inverted value of its documented contract since 1.1, silently disabling king safety in the middlegame and applying the endgame king PST from move 1. Fixing this one issue accounts for the largest measured Elo delta in any 1.5 development iteration (+224.37 Elo, n=1540 self-play).

The remaining gain comes from search refinements (LMP, SEE-aware capture ordering, IIR, countermove, razoring) and incremental eval improvements (knight outpost refactor, mopup guards). A planned king-safety overhaul (king attack ray scoring, pawn shelter/storm, piece tropism) was attempted in multiple forms but ultimately rejected -- the calibration of hand-set constants proved unreliable at this strength level. Better calibration tooling is needed before re-attempting these features.

Unchanged from 1.4 (logic, not just code): board representation, magic bitboards, move generation, NMP, RFP, futility, LMR, aspiration windows, PV array, repetition detection, PST values, mobility, open files, rook on 7th, bishop pair, pawn structure, UCI protocol, time management, threading.

Final NPS (release build, bench depth 18):

| Platform | 1.4 NPS | 1.5 NPS | Change |
|----------|---------|---------|--------|
| Linux (VM Debian, Ryzen 7 1700) | ~2,061K | ~1,620K | -21% |
| Windows 10 (native, Ryzen 7 1700) | ~2,188K | TBD | TBD |

The NPS drop is primarily attributable to SEE-aware capture ordering: every uncertain capture that could not be classified by the shortcut (`victim_value >= attacker_value`) requires a full SEE computation. The Elo gain of the feature (+54 self-play) dominated the per-node cost.

## Critical Bug Fixes

### `game_phase()` inversion

#### Problem

`game_phase()` is documented to return a value in [0, 256] where 256 means "full non-pawn material" (middlegame) and 0 means "bare kings" (pure endgame). Consumers use this directly:

- `king_safety()` multiplies its raw penalty by `phase_mg / 256`. With `phase_mg = 256` in middlegame, the full penalty applies. With `phase_mg = 0` in endgame, the penalty is zeroed.
- `pst_value()` for the king interpolates between `PST_KING_MG` and `PST_KING_EG` using `phase_mg`. Full middlegame uses MG; full endgame uses EG.

Since version 1.1, the function had its formula inverted: it counted *missing* material instead of remaining material. At startpos (no material missing relative to TOTAL_PHASE = 24) it returned 0; at bare kings (all material missing) it returned 256. The exact opposite of the contract.

Consumers, written assuming the contract, used the wrong tables:

- **King safety in the middlegame returned 0.** Every king-safety penalty was multiplied by `0 / 256 = 0`. The opponent could mass attackers on the king zone without consequence.
- **`PST_KING_EG` was used from move 1.** The endgame king PST encourages centralization (king toward d4/d5). In the opening, this gives the king a strong gradient toward the center -- where it gets attacked.

The bug had been silent through 1.1, 1.2, 1.3, and 1.4 because it interacted with multiple terms and the engine still played reasonably; it just played weaker than its evaluation knowledge suggested it should.

#### Fix

The fix inverts the counting. The new `game_phase()` starts at 0 and adds the phase contribution of each non-pawn piece on the board:

```cpp
static int game_phase(const Board& board) {
    int phase = 0;
    phase += popcount(board.piece_bb(KNIGHT)) * 1;
    phase += popcount(board.piece_bb(BISHOP)) * 1;
    phase += popcount(board.piece_bb(ROOK))   * 2;
    phase += popcount(board.piece_bb(QUEEN))  * 4;
    return std::min(phase, 24);
}
```

This matches the contract: startpos has 4N + 4B + 4R + 2Q = 4 + 4 + 8 + 8 = 24 (capped, returned as 256 after scaling). Bare kings = 0. Consumers are unchanged.

#### Impact

Self-play: +224.37 ±18.19 Elo (n=1540, LOS 100%). This is the single largest measured Elo improvement in any iteration of any Facon version.

The fix moves the engine into a new regime: king safety actually applies, opening-phase king centralization is no longer rewarded, and PST contributions reflect game phase correctly. Bench signature changed substantially (407.4M nodes at depth 18, up from 51.2M at depth 15 in 1.4 -- though the depth difference is the dominant factor).

### Drawn pawnless endings: bug diagnosed, fix attempted, fix reverted

#### Diagnosis

`evaluate()` has the following structure for handling mopup (pawnless winning endgames):

```cpp
bool no_pawns = board.piece_bb(PAWN) == 0;
if (no_pawns && std::abs(score) >= MOPUP_THRESHOLD) {
    Color strong_side = (score > 0) ? WHITE : BLACK;
    score += mopup_eval(board, strong_side);
}
```

`mopup_eval` correctly returns 0 for theoretically drawn pawnless endings (K+B vs K, K+N vs K, K+N+N vs K, K+B+B same-color vs K). The insufficient-material guard was extended in 1.3 (K+B vs K, K+N vs K) and 1.5 (K+N+N vs K, K+B+B same-color vs K).

But `score` already contained the strong side's material count BEFORE entering the if. A lone knight is 320cp, a lone bishop is 330cp, two same-color bishops 660cp. When `mopup_eval` returns 0, that material count remains as the final score.

Concrete example: in the position `8/8/5k2/3n1P2/6PK/8/8/8 b - - 0 1` (K+N vs K with two blocked white pawns), Facon reaches search depth 30+ with a score around +420 to +437 cp for the side with the knight, planning lines that capture both white pawns and reach pure K+N vs K. Other engines correctly evaluate this position as 0.00 (drawn). Facon treats the drawn KN-vs-K ending as a +3.2 to +4.4 pawn advantage.

#### Attempted fix

An override was implemented inside the threshold check: detect the case where `mopup_eval` returns 0 and force the final score to 0 instead of letting the material count stand. The change mirrored the same logic in `evaluate_verbose()` so the "eval" command would also report 0 for these positions.

#### Reversion

Extended testing showed the fix to be technically correct but competitively worse. In the same gauntlet field used in earlier measurements, the build with the override scored ~30 Elo below the build without it. Investigation via `go infinite` analysis on the example position confirmed the mechanism: the build without the override evaluates the position as winning (the misleading +420cp), reaches a stable bestmove early, and actively converts middlegame and complex endgame phases that lead toward such drawn finals. The build with the override evaluates the same position as 0, but the score propagates back through search so that intermediate positions on the way are seen as less attractive than they actually are. The engine becomes "honest about a draw it could often avoid" -- and avoids the lines that would force the opponent into one. In competitive play, the wrong-but-motivated evaluation outperforms the right-but-resigned evaluation.

The override was removed for 1.5. The underlying limitation -- the engine doesn't truly understand these endings as draws, only their material-signature surface -- remains. Properly fixing it requires endgame recognition (per-material-signature scaling factors, or tablebase probing for ≤7-piece positions), planned for a future version. Until then, Facon will continue to evaluate KB vs K, KN vs K, KNN vs K, and KBB same-color vs K as ~+330 to ~+660 cp rather than 0. The bug is latent: real-game impact is minimal because such pure endings are reached very rarely, and the engine's behavior in positions that would lead to them is now empirically the best available with the current evaluation framework.

## Search Refinements

### Late Move Pruning (LMP)

In non-PV nodes at shallow depth, once enough legal quiet moves have been searched without raising alpha, additional quiet moves are unlikely to discover anything. They are pruned entirely (the move is skipped without a recursive search).

Per-depth thresholds in `LMP_TABLE`. Captures, promotions, and killers are exempt (they are tactically interesting and worth searching even late). PV nodes are exempt (we need to find the actual best, not just a good-enough move).

Combined with SEE-aware capture ordering for a single shipped iteration.

### SEE-aware capture ordering

In 1.4, captures were ordered with a single `ORDER_CAPTURE` constant: MVV-LVA modulated all captures, treated identically. Bad captures (SEE < 0) and good captures (SEE >= 0) competed in the same range.

In 1.5, the captures are split into two ordering tiers:

- **GOOD captures** (`SEE >= 0`): scored above killers and tried before quiets. These are favorable trades.
- **BAD captures** (`SEE < 0`): scored below history and tried last. These are unfavorable; the search is interested in them only after quiets fail.

A shortcut avoids the SEE cost when the result is obvious: if the attacker is no more valuable than the victim (`PIECE_VALUE[attacker] <= PIECE_VALUE[victim]`), the capture is GOOD regardless of recapture (PxQ, NxR, BxR, etc.). Only ambiguous captures call into `see()`.

Bench NPS dropped ~22% (the SEE function is non-trivial; every PxP type capture now invokes it). Elo gain dominated: +54.00 ±17.32 (n=2020).

### Internal Iterative Reductions (IIR)

When a non-PV node has no TT move at sufficient depth, its move ordering will be poor for the first move. Searching at full depth wastes work. Instead, the depth is reduced by 1 before searching: a shallower search will store a move in the TT, and the next visit to the position (e.g. via aspiration window re-search) benefits from move ordering.

Implementation: a single line at the top of the move loop. Conditions: `!pv_node && depth >= IIR_MIN_DEPTH && tt_move == MOVE_NONE`. The reduction is recursive (compounds with NMP, LMR, etc.).

Shipped together with countermove and razoring: +14.42 Elo (n=2020).

### Countermove heuristic

For each `(piece, to_square)` pair (the description of the opponent's previous move), the engine remembers the quiet move that produced a beta cutoff in the child position. The next time the opponent plays the same (piece, to_square) move, the countermove is ordered after killers (slot above history).

Implementation requires passing the previous move through the negamax call chain. A 12 × 64 table indexed by (piece, to_square) stores the countermove per pair. Updated on beta cutoffs by non-capture moves. Reset every search.

### Razoring

The dual of reverse futility pruning. At depth 1-2 in non-PV nodes, if `eval + RAZOR_MARGIN <= alpha`, the position is so far below alpha that even a margin of 250cp (pawn + rook) cannot save it in a shallow search. Drop directly into quiescence without searching.

Constants: `RAZOR_MAX_DEPTH = 2`, `RAZOR_MARGIN = 250cp`. Triggers only in non-PV nodes (PV nodes need accurate scoring even when far below alpha).

## Evaluation Refinements

### Knight outpost refactor

In 1.4, knights on outpost squares received a flat +20cp bonus. An outpost was defined as: rank 4-6 (relative to the knight's color), no enemy pawn can attack the square.

In 1.5, the flat constant is split into two graduated bonuses:

- `KNIGHT_OUTPOST_REACHABLE = 10cp`: square is safe from pawn attack but no friendly pawn defends it. Indicates a knight that could be dislodged by a piece exchange.
- `KNIGHT_OUTPOST_SUPPORTED = 25cp`: square is safe AND defended by a friendly pawn. A firmly anchored knight that cannot be easily removed.

Modest measured gain (+9.82 Elo, n=920, LOS 87%) plus a fix.

### Knight outpost `forward_mask` fix

A latent bug in the 1.4 outpost detection. The `forward_mask` used to find disqualifying enemy pawns (pawns that could potentially attack the outpost) incorrectly included the knight's own rank. An enemy pawn on the same rank as the knight in an adjacent file would disqualify the outpost -- but such a pawn cannot attack the knight (pawns capture diagonally forward, not horizontally).

Fix: `forward_mask` is now strictly ahead of the knight (relative ranks 1+ further forward).

Edge case: a white knight on rank 8 has no squares ahead. The naive code would compute `1ULL << 64` -- undefined behavior. Guarded explicitly: return no outpost in that case (white knights rarely sit on rank 8 anyway).

### Mopup insufficient material guard extended

In 1.3 (and corrected in 1.4), mopup evaluation was disabled when the strong side has exactly one minor piece (K+B vs K or K+N vs K) and the position is a theoretical draw. Without the guard, mopup would push the kings toward the corner in a drawn position, refusing draws and wandering.

In 1.5, the guard is extended to cover two more drawn material configurations:

- **K+N+N vs K** (two knights versus bare king): theoretical draw. The two knights cannot force mate against a king played correctly.
- **K+B+B vs K with same-colored bishops**: theoretical draw. Two bishops on the same color square act as a single bishop and cannot deliver mate.

K+B+B with opposite-colored bishops is correctly excluded from the guard -- this is a winning endgame and mopup is desired.

Implementation: check `popcount` of `KNIGHT`, `BISHOP` bitboards for the strong side, plus light-/dark-square classification when there are two bishops.

## Transposition Table -- Aging

### Problem

In 1.4, replacement was depth-preferred only: overwrite if same position, empty slot, or new depth >= stored depth. Deep entries from earlier positions in the game persist indefinitely. After many moves, the TT fills with stale deep entries that can never be replaced by fresh shallow searches at the current root.

### Solution

A global `current_generation` counter is bumped at the start of each search (modulo 64; the counter wraps). Each TT entry stores the generation it was written with, packed into the bound byte:

```cpp
uint8_t gen_bound;  // (generation << 2) | bound (BOUND_NONE/UPPER/LOWER/EXACT)
```

Three bits remain unused; both fields fit in one byte. Entry size unchanged at 16 bytes.

Replacement compares `(age_in_generations * 4) + (stored_depth - new_depth)` against zero. An entry one generation old is treated as 4 plies shallower for replacement purposes. A two-generation-old depth-15 entry can be replaced by a fresh depth-8 entry. A current-generation depth-15 entry blocks a fresh depth-8.

### Impact

Difficult to measure directly because the cost is "stale entries blocking fresh data" -- the impact is invisible in bench (no game history). Measured gain in self-play at 16MB hash: ~+5-15 Elo. The gain scales inversely with hash size (with infinite hash, aging would have zero effect).

## Infrastructure

### `bench` default depth raised to 18

In 1.4, the bench default was depth 15 with 51M nodes. At Facon's current strength, depth 15 measurements are noisy because individual positions resolve in single iterations. Depth 18 provides more representative measurements; positions take meaningful time and per-position node counts span 1-2 orders of magnitude.

Bench signature at depth 18: **299,881,540 nodes** (deterministic; used as a no-regression check across builds).

### `bench` position rebalancing

In 1.4, position 3 took ~37% of bench time at depth 12 while position 9 took ~0.03%. At depth 18 the imbalance worsens. In 1.5 we replaced 6 of the 10 positions (kept 2, 4, 5, 6) and selected new positions targeting specific search features:

| # | Label |
|---|-------|
| 1 | Opening, Italian Game with c3 -- baseline NPS, TT cold-cache |
| 2 | Middlegame, ~35 legal moves -- move ordering, LMR, history |
| 3 | Middlegame with hanging exchanges -- qsearch, SEE x-ray |
| 4 | Middlegame with material advantage -- NMP effectiveness |
| 5 | Middlegame closed structure -- NMP zugzwang guards, futility |
| 6 | Middlegame king attack -- king safety zone evaluation |
| 7 | Middlegame with pawn structure tension -- pawn eval |
| 8 | Middlegame with checking sequences -- check extensions, tactics |
| 9 | Endgame R+2P vs R -- deep search, passed pawn dynamics |
| 10 | Endgame KBN vs K -- mopup corner-distance evaluation |

Each position is paired with a label in the `BenchPosition` struct printed during `bench verbose`. At depth 18, per-position share ranges 1.6% (position 9, rook endgame) to 22.5% (position 5, closed structure).

### `bench verbose` flag

Default `bench` runs quietly: each position prints a one-line summary (nodes, ms). `bench verbose` enables full UCI output (`info depth ...`, `info currmove`, TM messages). Useful for diagnosing whether a regression is real or noise.

### `bench TM.infinite = true` during benchmark

The bench used to run with whatever TM state was previously configured. If the user had been using `go wtime ... btime ...`, the TM clock state could interfere with bench search depth. Now the bench explicitly forces `TM.infinite = true` before each position, restoring the previous state afterward.

### `bench` short-circuit on mate-in-1

A few bench positions can have a mate-in-1 at root (depending on position selection). Without a guard, the bench logs a "depth 0" line and continues iterating to the configured depth -- wasting time. Now the iteration loop exits cleanly when a mate is found at depth 1.

### `eval` UCI command

New non-UCI debug command. Prints a per-component breakdown of `evaluate()` for the current position. Useful for tuning eval constants, diagnosing why the engine prefers a particular move, and verifying that recently-changed eval code does what was intended.

```
> eval
Static evaluation (cp = centipawns, +ve favors White unless noted):
  game phase (0 = bare kings, 256 = full non-pawn material): 256
                         White       Black       Diff (W - B)
  Material:             4000        4000           0
  PST:                   -95         -95           0
  King safety:             0           0           0
  ...
```

Implementation: `evaluate_verbose()` in `eval.cpp` mirrors the structure of `evaluate()` but accumulates each component separately and prints them.

### `evaluate_verbose()` label clarifications

The eval output had two confusing labels in early drafts:

- "game_phase: 0 = endgame, 256 = middlegame" -> "0 = bare kings, 256 = full non-pawn material". The new label is factual (phase counts non-pawn material) rather than interpretive (the engine has no formal "opening/middlegame/endgame" labels; everything is continuous in phase).
- The mopup "skipped: |partial| below threshold" became "position not winning enough"; "insufficient material in mopup" became "insufficient material to win". The new strings describe the situation rather than the internal mechanism.

### `cmd_setoption()` Hash parse safety

`cmd_setoption()` was using `std::stoi(value_str)` to parse the Hash size. Release builds use `-fno-exceptions`, so `std::stoi` on a non-numeric string calls `std::terminate()` -- crashing the engine. A malformed UCI command like `setoption name Hash value abc` would kill the engine instead of producing an error message.

Fix: replaced with a manual digit-parse pattern (same as `cmd_perft()`). Returns silently on invalid input rather than crashing.

### Final summary on aborted iteration

When the search is interrupted mid-iteration (hard time limit hit, `stop` command), several minutes of search activity could elapse silently between the last heartbeat (every 5 minutes) and the `bestmove` output. The log showed the heartbeat from minutes earlier and then jumped straight to the bestmove -- the cumulative nodes searched, final NPS, and exact elapsed time during that final partial iteration were lost.

The search now emits two lines right before returning, but only when the iteration in progress was actually interrupted (`abort_search_ == true`):

1. A UCI-parseable `info depth` line with the depth at which the search was aborted (`last_depth + 1`), total nodes, NPS, elapsed time, and hashfull. Raw units. GUIs read this for their displays.
2. An `info string aborted while searching at depth N -- M nodes, h:mm:ss,ms, M nps -- using score from depth N-1: ±Xcp` line. Human-readable summary. GUIs ignore info strings; PGN annotators and log readers see the summary.

When the search ends cleanly at iteration boundary (TM `soft_stop()` after a completed iteration), no summary is emitted -- the last regular `info depth` line already reflects the complete state.

The score reported in the summary is always the score from the last completed iteration. Partial iterations are not trustworthy and the loop discards their scores already; the summary preserves that contract.

### Command-line `bench` mode

The binary can now be invoked directly as:

```
./facon-1.5 bench
./facon-1.5 bench verbose
./facon-1.5 bench depth 22
./facon-1.5 bench verbose depth 22
```

The benchmark runs once with the same flag handling as the UCI `bench` command (the implementation is a pure wrapper that constructs an `std::istringstream` from the argument string and dispatches to the existing `cmd_bench()` handler). The banner and TT info are still printed when stdin is a terminal, same as in interactive UCI mode. After the bench completes, the binary exits with code 0.

This is useful for automated NPS regression tests, CI runs, profiling sweeps over compilation flags (the analysis that produced the `-fomit-frame-pointer` decision was done this way), and the development validation pattern of "compile, run bench, check signature against last known value". It removes the friction of having to feed `bench\nquit\n` to stdin every time.

Unknown command-line arguments produce a usage message and exit code 1. Only `bench` is currently supported.

### `currmove` output suppressed for first 2 seconds

UCI-standard `info depth N currmove M currmovenumber K` lines are emitted from the root before searching each move. At shallow depths (say, depth 5 to 12) each move at the root completes in well under a millisecond, producing hundreds of currmove lines per second. In testing under Arena's tournament mode on Windows, this volume of stdout traffic could saturate the GUI's pipe-reader event loop in some cases, with knock-on effects on observed search throughput.

The fix is a single time gate at the emission site: `currmove` lines are now suppressed during the first 2 seconds of each search. After 2 seconds, currmove output resumes normally. By that point the search depth has typically grown enough that each move at the root takes long enough to be worth reporting, and the per-second line rate is moderate.

Heartbeats, `new best` lines, and final iteration summaries are unaffected -- they are emitted on different conditions and were never the source of the volume. Bench output is also unaffected -- the bench position search profile rarely lingers near the early shallow depths long enough to be in the suppressed window for any individual position.

### Score sign convention cleanup

The `+` prefix on positive scores in info string output (AW fail-low/high, "new best" annotations, and the new final summary) was previously applied with the condition `score >= 0`, which produced `+0cp` for exact-zero scores -- inconsistent with most other engines and slightly confusing visually. The condition is now `score > 0`, so zero scores print as `0cp` without a sign.

Affects four call sites in `search.cpp`. The behaviour change is purely cosmetic; PGN annotators and log parsers that match the score field already handle both signed and unsigned numeric tokens.

## Build Changes

### MSVC support removed

The 1.4 (and earlier) `CMakeLists.txt` contained a branch for native MSVC builds with the appropriate `/O2 /Ob2 /GL /arch:SSE4.2` flags. The branch was never functional: `bitboard.h` uses GCC/Clang builtins directly (`__builtin_popcountll`, `__builtin_ctzll`, `__builtin_clzll`), which MSVC does not understand. A native MSVC build would have failed at compile time on those three lines.

The branch existed as a leftover from initial scaffolding (Facon 1.0). It was kept through subsequent releases under the assumption that "we'll add wrappers eventually". Adding MSVC intrinsic wrappers (`__popcnt64`, `_BitScanForward64`, `_BitScanReverse64`) is straightforward conceptually, but introducing untested code to a release that the developer cannot validate (no Windows MSVC machine on hand) violates the project's "the binary tested is the binary published" principle.

The MSVC branch has been removed entirely. The build system now declares an explicit platform set (GCC, Clang, MinGW-w64) and rejects any other compiler at configure time with `FATAL_ERROR`. The error message points users toward MinGW-w64 for native Windows builds.

Side effect: the `_CRT_SECURE_NO_WARNINGS` macro definition in the Windows block was also removed (it was an MSVC-specific suppression of "deprecated" warnings on libc functions like `strcpy`; MinGW's GCC does not emit those warnings).

### `-fomit-frame-pointer` added to Release builds

A deliberate sweep over GCC tuning flags during 1.5 release prep measured the effect of various combinations on bench NPS. Result table (Ryzen 7 1700, Linux, single run per variant):

| Variant | NPS | Δ vs. base |
|---------|----:|-----------:|
| `-march=native` | 1,668,214 | +3.45% |
| `-march=x86-64-v3` | 1,651,484 | +2.41% |
| `-fomit-frame-pointer` (alone, over base) | 1,649,803 | +2.30% |
| O3 + LTO only ("clean") | 1,638,670 | +1.62% |
| O3 + LTO, no `-funroll-loops` | 1,629,029 | +1.02% |
| `-march=x86-64-v2` | 1,618,811 | +0.38% |
| `-Ofast` + various ("aggressive") | 1,617,013 | +0.27% |
| Previous 1.4 baseline | 1,612,647 | -- |

Decision: add **`-fomit-frame-pointer` only** to the Release flag set.

Rejected for various reasons:
- `-march=x86-64-v3`: gains +2.41% but drops compatibility with CPUs pre-2013 (pre-Haswell Intel / pre-Zen 1 AMD). Not worth the compat loss for a public release.
- `-march=native`: not distributable -- the resulting binary may not run on other CPUs.
- `-Ofast`: enables `-ffast-math` which breaks IEEE 754 floating-point semantics. Facón is integer-heavy, so the gain is marginal; the risk of subtle FP-related bugs is not.
- Removing `-funroll-loops`: marginal NPS difference, no compelling reason.

The single `-fomit-frame-pointer` flag delivers ~70% of the maximum measured improvement at zero compatibility cost, which is the right trade-off for a public release.

A future opt-in `-DCPU_LEVEL=v3` option may be added for users with modern CPUs willing to trade compatibility for performance; that was deferred to avoid scope creep on 1.5.

## Features Attempted and Rejected

Several features were attempted during 1.5 development and rejected because measurement showed neutral or negative Elo. This section documents them as guidance for future development and as a record that the work was done.

### King attack overhaul (rejected, n=20 + n=80 partial)

The 1.1-era king safety scored binary attacker presence per piece type with quadratic scaling and no upper bound. In 1.5 we attempted to replace this with:

- Per-square attack weighting (a queen attacking 4 zone squares contributes 4x weight, not 1x).
- A multi-attacker bonus when distinct piece types coordinate.
- A capped penalty curve (linear + quadratic with a hard cap, since the pure quadratic without a cap produced penalties of 4,800-10,000 cp in worst cases).

Without the cap: -356 Elo. With a cap (200 cp): -107 Elo at n=20 (a fair improvement but still strongly negative). Rejected.

The cause is calibration: the relative weights of "knight attacker", "queen attacker", "multi-attacker bonus", and the penalty curve coefficients are inter-dependent and hand-set values produced inconsistent results. Better calibration tooling is needed before re-attempting this feature.

### Pawn shelter / storm (rejected, n=240)

For flank kings, a shelter bonus rewarded an intact pawn shield (closest own pawn on king's file or adjacent files) and a storm penalty punished advancing enemy pawns toward the king. Used in many HCE engines.

Result: -87.20 ±36.90 Elo at n=240. Rejected.

The most likely cause: the missing-pawn penalty (-25cp per file with no own pawn ahead, max -75cp) is too aggressive for this engine's playing style. The engine moves pawns more freely than the shelter-bonus model assumes, so legitimate strategic pawn advances trigger large false-negative penalties.

### Piece tropism (neutral, n=640)

Bonus for own pieces close to the enemy king. Indexed by Manhattan distance with per-piece-type tables (knight, bishop, rook, queen).

Result: -0.00 ±21.04 Elo at n=640. Exactly neutral.

The 30-40 cp maximum total contribution per side is dwarfed by other eval terms (PST ±50 cp, king safety ±0-300 cp, pawn structure ±20-50 cp). The motor's decisions are dominated by the larger terms. Either remove tropism entirely or scale it 3-5x larger -- but without tuning, larger values introduce noise.

### Singular extensions (neutral, n=1760)

When a TT move appears clearly better than all alternatives at sufficient depth, search it one ply deeper. The verification search runs at reduced depth with the TT move excluded; if every other move falls below `tt_score - margin * depth`, the TT move is singular.

Implementation required adding an `excluded_move` parameter to `negamax()`, skipping that move in the move loop, and special-casing TT writes and mate detection when `excluded_move` is set.

Result: -0.59 ±11.77 Elo at n=1760. Neutral with extremely tight CI.

Singular interacts with move ordering -- if the TT move is genuinely best, the engine is already searching it first and the +1 ply extension can only add nodes, not change the bestmove. With perfect move ordering, singular is redundant. With imperfect ordering, singular helps. The 1.5 outcome suggests Facon's move ordering is good enough that singular doesn't help in measurable amounts. A re-test after move ordering improvements is worth doing in a future version.

### Other partial attempts

PVS (Principal Variation Search), two variants attempted, both neutral. The implementation is conceptually straightforward but interacted poorly with the existing aspiration window + LMR.

## Known Limitations

- **Hand-tuned eval constants**: all eval weights are set by hand based on empirical self-play measurements. This was the principal limitation revealed during 1.5 development -- several conceptually sound features (king attack overhaul, pawn shelter/storm, piece tropism) failed to gain Elo because of calibration, not concept. Better calibration tooling is a major item for future development.
- **No king attack overhaul**: the 1.1-era king safety zone (binary attacker presence per piece type with quadratic scaling) remains in 1.5. Attempts to replace it with per-square attack scoring failed in measurement. To be re-attempted in a future version once better eval calibration tooling is in place.
- **No pawn shelter/storm evaluation**: same situation as above -- planned but pending better calibration.
- **Drawn pawnless endings evaluated as material**: KB vs K, KN vs K, KNN vs K, and KBB same-color vs K are theoretically drawn but `evaluate()` returns the raw material count (~+330 to ~+660 cp) rather than 0. An override that forced 0 in these cases was attempted but reverted -- although technically correct, it caused measurable gameplay regression by demotivating the engine from reaching positions whose drawn final endings propagated back through search. Proper resolution requires material-signature endgame recognition or Syzygy tablebase probing, both planned for a future version.
- **No endgame recognition**: K+R vs K+B and similar endings where the material advantage is below `MOPUP_THRESHOLD` (300cp) have no positional direction. The engine holds the advantage but may not make progress. Planned for a future version.
- **No Syzygy tablebases**: endgame play relies entirely on search depth + mopup. Planned for a future version.
- **Single-threaded**: no Lazy SMP. Planned for a future version.
- **No incremental eval**: full evaluation re-computed at every node. Mild speedup available by incrementally updating PST values in `make_move`/`unmake_move`. Planned for a future version (pure speedup, no eval logic change).
