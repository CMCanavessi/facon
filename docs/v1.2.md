## Overview

Facón 1.2 "Rojo Vivo" focuses on search pruning (NMP), endgame technique (mopup), and engine observability. The hash corruption bug fixed in pre-work was latent since 1.0 and had broad impact on TT effectiveness and repetition detection — its fix alone is expected to contribute measurable Elo independent of the new features.

Unchanged from 1.1: board representation, magic bitboards, move generation, killer heuristic, king safety evaluation, time management model, UCI option handling.

## Pre-work Bug Fixes

### `unmake_move()` hash corruption

**Root cause**: `hash = st.hash` was placed *before* the piece operation calls (`move_piece`, `put_piece`, `remove_piece`). Each of these functions XORs the hash incrementally as a side effect. So the sequence was:

```
hash = st.hash           // restore to H_before
move_piece(to, from)     // hash ^= piece[from] ^ piece[to]  ← corrupts hash
put_piece(captured, to)  // hash ^= captured[to]             ← corrupts further
```

The board hash after unmake was `H_before XOR piece_delta` instead of `H_before`. Since `H_before` was always the correct pre-move hash (saved correctly in `make_move()`), the fix is to move `hash = st.hash` to the **last line** of `unmake_move()`, after all piece operations. Their hash side-effects are then overridden by the saved value.

**Symptoms**: the root hash alternated between two values across iterative deepening iterations for the same position. TT hit rates were reduced (lookups used the wrong hash). Repetition detection in the PV walk was unreliable. The `"PV continues after threefold repetition"` GUI warning was a downstream symptom.

**Scope**: `unmake_move()` in `board.cpp`. `unmake_null_move()` was unaffected (it restores `hash = st.hash` and performs no piece operations).

### `make_null_move()` full_move_number corruption

**Root cause**: `make_null_move()` flipped `side_to_move` but never incremented `full_move_number`. Meanwhile, `unmake_null_move()` decremented it when `side_to_move == BLACK` after the flip, mirroring `unmake_move()`. Every null move by Black produced a decrement without a prior increment.

**Fix**: added `if (side_to_move == WHITE) full_move_number++;` in `make_null_move()` after the side flip (at that point `side_to_move` is already the new side, so `WHITE` means Black just moved — matching `make_move()` logic).

**Impact**: corrupted `full_move_number` fed into phase calculation (material-based interpolation between MG/EG eval), which fed into time management. Effect was small per move but accumulated over long games with deep NMP trees.

### Double `generate_all_moves()` in TT probe path

**Root cause**: when the TT probe returned a `tt_move`, the code called `generate_all_moves()` to validate it against the legal move list, then called `generate_all_moves()` again a few lines later for the main search loop.

**Fix**: moved a single `generate_all_moves()` call before the TT probe. The same list is used for TT move validation and the main loop.

**Impact**: ~2-5% nps improvement.

## Search — Null Move Pruning

### Concept

If we skip our turn (make a "null move") and the opponent still cannot improve beyond beta at reduced depth, our position is so strong that the opponent won't be able to refute it — prune immediately.

### Implementation

```cpp
// In negamax(), after TT probe and before move loop:
if (do_null && !in_check && ply > 0
    && depth >= NMP_MIN_DEPTH
    && (board.piece_bb(side) & ~board.piece_bb(side, PAWN)).count() > 1)
{
    board.make_null_move();
    Score null_score = -negamax(board, -beta, -beta+1,
                                depth - 1 - NMP_REDUCTION, ply+1, false);
    board.unmake_null_move();

    if (null_score >= beta)
        return beta;
}
```

Constants: `NMP_MIN_DEPTH = 3`, `NMP_REDUCTION = 3`.

### Guards

| Guard | Reason |
|-------|--------|
| `do_null` | Prevents consecutive null moves (passed as `false` to child) |
| `!in_check` | Null move is illegal when in check |
| `ply > 0` | Never at root — root needs a real move for `bestmove` |
| `depth >= NMP_MIN_DEPTH` | Avoid at shallow depths where reduction yields depth ≤ 0 |
| non-pawn material | Avoids zugzwang: in pure pawn endings, passing the turn is often fatal |

### `in_check` hoisting

`in_check` is computed once before NMP (required for the guard) and reused for the mate/stalemate detection after the move loop. Previously computed twice.

## Search — Triangular PV Array

### Concept

Instead of reconstructing the PV from TT entries at the end of search (fragile — entries can be overwritten mid-search), maintain an explicit triangular array where each ply stores its own PV line.

```
ply 0: [m0, m1, m2, m3]       ← full PV from root
ply 1:     [m1, m2, m3]       ← PV from ply 1
ply 2:         [m2, m3]       ← PV from ply 2
```

### Data structures

```cpp
// search.h
Move pv_table_[MAX_PLY][MAX_PLY];
int  pv_length_[MAX_PLY];
```

### Update rule

Only updated when alpha is raised (score > alpha):

```cpp
pv_table_[ply][0] = m;
for (int j = 0; j < pv_length_[ply+1]; j++)
    pv_table_[ply][j+1] = pv_table_[ply+1][j];
pv_length_[ply] = pv_length_[ply+1] + 1;
```

`pv_length_[ply] = 0` is set at the top of each `negamax()` call. `memset(pv_length_, 0)` resets the whole array at the start of each ID iteration.

## Search — PV Repetition Detection

### Problem

The PV can contain moves that cycle through positions already seen in the game — a draw by repetition that the engine correctly evaluates as 0 but that produces a misleading PV line in GUIs.

### Root cause (diagnosed via debug)

The earlier `is_repetition()` approach failed because the PV walk operates on a fresh board copy and `is_repetition()` scans `history[]` relative to that copy's `history_ply`. Positions from the actual game (before the root) are present in `history[]` but the alternating hash corruption (pre-work fix) caused hash mismatches. After fixing the hash bug, the `seen[]` approach works correctly.

### Implementation

```cpp
// In go(), PV printing block:
uint64_t seen[MAX_GAME_HISTORY + MAX_PLY + 2];
int      seen_count = 0;

// Seed with full game history (hashes before each move)
for (int i = 0; i < pv_board.history_ply; i++)
    seen[seen_count++] = pv_board.history[i].hash;
seen[seen_count++] = pv_board.hash;  // root position

for (int i = 0; i < pv_length_[0]; i++) {
    Move pv_move = pv_table_[0][i];
    if (!pv_board.is_legal(pv_move)) break;

    pv_board.make_move(pv_move);

    // Check AFTER make_move — we're checking the resulting position
    bool repeated = false;
    for (int j = 0; j < seen_count; j++)
        if (seen[j] == pv_board.hash) { repeated = true; break; }
    if (repeated) break;  // stop WITHOUT printing this move

    seen[seen_count++] = pv_board.hash;

    // Print only after confirming no repetition
    std::cout << ' ' << uci_move_str(pv_move);
}
```

Key detail: `make_move()` then check, then print. Printing before the check was the original bug — the move that caused the repetition was already output before detection.

## Evaluation — Mopup

### Concept

In pawnless endings where one side has a decisive material advantage, standard PSTs don't guide the winning king to corner the losing king. Mopup adds an explicit bonus to help convert technically won positions.

### Activation condition

```cpp
if (board.piece_bb(PAWN) == 0 && std::abs(score) >= MOPUP_THRESHOLD)
```

`MOPUP_THRESHOLD = 300` (roughly a minor piece advantage).

### Formula

```cpp
int mopup_eval(const Board& board, Color strong_side) {
    Square strong_king = board.king_square(strong_side);
    Square weak_king   = board.king_square(~strong_side);

    int corner_bonus    = MOPUP_CORNER_WEIGHT * center_distance(weak_king);
    int proximity_bonus = MOPUP_PROXIMITY_WEIGHT * (14 - king_distance(strong_king, weak_king));

    return corner_bonus + proximity_bonus;  // from White's perspective
}
```

- `center_distance(sq)`: Manhattan distance from the nearest central square (0 = center, max 6 = corner)
- `king_distance(a, b)`: Manhattan distance between two squares (max 14)
- Result added to score from White's perspective, then re-oriented by `evaluate()`

Constants: `MOPUP_CORNER_WEIGHT = 10`, `MOPUP_PROXIMITY_WEIGHT = 5`.

## Observability

### currmove / currmovenumber

At ply 0 with `depth > 1`, each root move emits a standard UCI line as it begins searching:

```
info currmove e2e4 currmovenumber 1
info currmove d2d4 currmovenumber 2
...
```

This is gated to `depth > 1` to avoid spamming at depth 1 where root moves are evaluated near-instantly.

### New-best SAN info string

Whenever a new best move is found at the root (alpha raised inside the root move loop), an `info string` line is emitted in human-readable SAN notation:

```
info string new best: Nf3+ [3/20]  score cp 42  depth 8
```

The `[3/20]` field is `[move_index / root_move_count]`. Requires `move_to_san(Move m) const` in `board.cpp` — full SAN generation with disambiguation (file, rank, or both for ambiguous pieces), castling (`O-O`, `O-O-O`), promotions (e.g. `e8=Q`), and check/checkmate markers (`+`, `#`).

This output is suppressed when `!isatty(STDIN_FILENO)` since it is informational only and GUIs do not parse it.

### Heartbeat

If `HEARTBEAT_INTERVAL_MS` (5 minutes = 300 000 ms) have elapsed since the last output, a heartbeat is emitted:

```
info string heartbeat depth 14  time 312450ms
```

Controlled by `last_output_ms_` (updated on every `info depth ...` line and on every new-best SAN emission). Distinguishes a crash from a deep search at long TC without requiring external tooling.

## Time Management

### `extend_time()` reason parameter

```cpp
// timeman.h
void extend_time(double factor, const char* reason = "");
```

When `reason` is non-empty, emits an info string before applying the extension:

```
info string TM extend ×1.50 (PV change)
info string TM extend ×1.25 (score drop)
```

Call sites in `search.cpp`:
- `TM.extend_time(1.5, "PV change")` — PV move changed between iterations
- `TM.extend_time(1.25, "score drop")` — score dropped ≥30 cp

### `start()` allocation report

On each call to `TM.start()` (once per `go` command), emits:

```
info string TM alloc: soft=4200ms hard=8400ms
```

Shows the computed soft and hard limits for the current move. Useful for diagnosing time management behavior at long time controls and verifying the engine allocates reasonable time per move.

## Infrastructure

### UCI threading

`cmd_go()` captures `board_` by value and launches the search in `search_thread_`. The UCI loop returns immediately. `cmd_stop()` sets `TM.stop = true` and joins the thread. The `quit` command also joins before exiting.

Previously `stop` was processed only after the search finished (which could be never on `go infinite`).

### `isatty()`-gated output

The startup banner, `TT.print_info()`, and the interactive `> ` prompt are printed only when `isatty(STDIN_FILENO)` returns true. When launched by a GUI or fastchess, stdout is clean UCI. When run interactively in a terminal, the full human-readable output is shown.

### TT silent constructor

`TT` is a global object; its constructor runs before `main()`. The constructor now calls `resize(mb, silent=true)`. `print_info()` is called explicitly in `main()` after the banner, gated behind `isatty()`. Previously the TT size message appeared before the banner.

## Post-release Bugfixes

These three bugs were discovered during a post-release code audit and fixed before the official v1.2 tag. None affect the correctness of the best move selected; all affect either time responsiveness or output accuracy.

### Quiescence time check

`quiescence()` checked `(stats_.nodes & 2047) == 0` to decide when to call `TM.should_stop()`. The problem is that `stats_.nodes` is only incremented in `negamax()`, not in `quiescence()`. During a deep tactical sequence — many recursive capture-search calls without returning to negamax — `stats_.nodes` is frozen, so the condition evaluates to the same value (true or false) on every call in that chain.

In the overwhelming majority of entries to quiescence, `stats_.nodes` is not a multiple of 2048, so the condition is permanently false for the entire chain. The time check is effectively disabled until the chain terminates and control returns to negamax.

**Fix**: switch to `stats_.qnodes & 2047` in `quiescence()`. `stats_.qnodes` increments on every quiescence call, so the check fires regularly regardless of how deep the capture sequence goes.

**Practical impact**: with `STOP_GRACE_MS = 100ms` in the hard limit, losing a few seconds of time checks in a deep capture chain is unlikely to cause a timeout in the gauntlet. At VVLTC (12h+10min) the risk is even lower — the hard limit is much more generous. No gauntlet losses to this bug are suspected.

### Nodes / NPS reporting

All three places that report nodes and NPS — the end-of-iteration UCI info line, the heartbeat, and the new-best info string — were using `stats_.nodes`, which only counts negamax nodes. `stats_.qnodes` (quiescence nodes) was never included.

In tactical positions quiescence can generate 5–10× more nodes than the main search. The reported node count and NPS shown in Arena and other GUIs were therefore severely underestimated throughout all of 1.2's testing.

**Fix**: all three output sites now use `stats_.nodes + stats_.qnodes`. The `SearchStats` comment is also corrected: `nodes` is now documented as negamax-only, with a note to use the sum for reporting.

**Practical impact**: zero effect on playing strength. Affects only the accuracy of the numbers shown in GUIs and used for performance comparisons between versions.

### PV seen[] array size

The repetition-detection array in the PV walk was declared as `seen[MAX_GAME_HISTORY + MAX_PLY]` = `seen[1152]`. The seed pass fills one entry per `history_ply` plus one for the root position (up to 1025 entries). The walk then adds one entry per PV move (up to 128). Worst case: 1153 entries in a 1152-element array.

The existing overflow guard `if (seen_count < MAX_GAME_HISTORY + MAX_PLY)` prevented an actual out-of-bounds write, but it silently skipped registering the last position in the array. If that exact position appeared again later in the walk, the repetition would not be detected.

**Fix**: declare `seen[MAX_GAME_HISTORY + MAX_PLY + 2]`. The +2 gives one slot of true slack beyond the worst case, and the overflow guard is now a genuine safety net rather than a functional constraint.

**Practical impact**: requires a game of 512+ half-moves with a PV of 128 moves simultaneously to trigger — impossible in normal play. Zero expected impact on any gauntlet result.

## Known Limitations

- **Time management not yet phase-aware**: the soft limit does not scale with game phase. Deferred to 1.3 — will be evaluated once NMP depth/extension behavior is characterized from the 1.2 gauntlet.
- **`depth <= 0` in negamax**: the quiescence entry condition uses `depth <= 0` rather than `depth == 0`. Harmless without LMR, but will require `depth == 0` + `new_depth = std::max(0, reduced_depth)` when LMR is added in 1.3.
- **`is_repetition()` detects twofold, not threefold**: returns true on the second occurrence of a position (not the third). This is the standard behavior for search purposes (if the position can repeat, evaluate as draw), but worth noting.
