## Overview

Facón 1.1 "Herrumbre" is an incremental improvement over 1.0 "Óxido". The core architecture — bitboard representation, magic bitboards, Zobrist hashing, move encoding, move generation, transposition table, and UCI protocol — is unchanged. The improvements are concentrated in three areas:

- **Search**: killer move heuristic, seldepth tracking
- **Evaluation**: king safety penalty
- **Time management**: soft/hard limit model with dynamic extension
- **Stability**: abort flag, root move tracking, `is_legal()` piece ownership check, TT move validation, TT pollution prevention

Measured improvement: **+120 Elo** over 1.0 (gauntlet testing vs field Elo ~1358, 1040 games at 2min+1sec).

The engine pipeline is unchanged:

```
UCI input → position setup → move generation → search → evaluation → UCI output
```

## Search

### Iterative Deepening

Unchanged in structure. The main addition is the `abort_search_` flag and `root_best_move_` tracking, documented in [Stability Fixes](#stability-fixes).

After each completed iteration, the engine now checks `soft_stop()` instead of `should_stop()`. This allows dynamic time extensions based on search feedback — see [Time Management](#time-management).

If the search was aborted mid-iteration (`abort_search_` is true) and we are past depth 1, the result is discarded and the last fully completed iteration is used.

### Negamax Alpha-Beta

Unchanged in structure. Key difference: **the search never does an early TT return at the root** (`ply == 0`). The root always searches all moves so that `root_best_move_` is set from the actual move loop. See [Stability Fixes](#stability-fixes).

### Draw Detection, Checkmate, Stalemate, Quiescence Search

Unchanged from 1.0. See [v1.0.md — Search](v1.0.md#search).

### Move Ordering

1.1 adds a third ordering tier: **killer moves**.

| Tier | Score | Description |
|------|-------|-------------|
| TT move | 1,000,000 | Best move from a previous search at this position |
| Captures | 100,000 + MVV-LVA | Most Valuable Victim / Least Valuable Attacker |
| Killer 1 | 90,000 | Most recent quiet move that caused a beta cutoff at this ply |
| Killer 2 | 80,000 | Second killer move at this ply |
| Quiet moves | 0 | No further ordering in 1.1 |

#### Killer Move Heuristic

A killer move is a quiet move (non-capture, non-promotion, non-en-passant) that caused a beta cutoff. If a quiet move was good enough to cause a cutoff at this ply in a sibling node, it is likely to be good here too — the position is similar enough that the same threat probably applies.

Two killer slots are maintained per ply. When a new killer is stored:

```cpp
void Search::store_killer(Move m, int ply) {
    if (m == killers_[ply][0]) return;  // Already stored
    killers_[ply][1] = killers_[ply][0]; // Shift older killer to slot 1
    killers_[ply][0] = m;                // Store new killer in slot 0
}
```

Killers are indexed by ply (distance from the root), not by depth. This is correct because killers represent threats in a specific tactical context, which is determined by how many moves have been played from the root.

Killers are cleared at the start of each `go()` call. Killers from a previous position have no relevance and can mislead move ordering.

#### TT Move Validation

Before using the TT move for move ordering, it is verified against the generated move list:

```cpp
if (tt_move != MOVE_NONE) {
    MoveList pseudo;
    generate_all_moves(board, pseudo);
    bool found = false;
    for (int i = 0; i < pseudo.count; i++)
        if (pseudo.moves[i] == tt_move) { found = true; break; }
    if (!found) tt_move = MOVE_NONE;
}
```

This guards against hash collisions producing a move that is not even pseudo-legal in the current position. Without this check, such a move would be assigned `ORDER_TT_MOVE = 1,000,000` and searched first, wasting a full subtree search.

### seldepth Tracking

`stats_.seldepth` tracks the maximum ply reached across all calls to `quiescence()` during the current iteration. It is reset to `depth` at the start of each iteration (so seldepth is always at least as large as depth) and updated whenever a deeper ply is reached:

```cpp
if (ply > stats_.seldepth) stats_.seldepth = ply;
```

The value is reported in the UCI info line:

```
info depth 8 seldepth 23 score cp 42 nodes 148032 ...
```

A large seldepth (much greater than depth) indicates that the quiescence search is extending significantly into tactical lines. For example, `depth 8 seldepth 23` means the main search went 8 plies deep but quiescence continued up to 15 more plies in the most tactical lines found.

## Evaluation

### Material Values and Piece-Square Tables

Unchanged from 1.0. See [v1.0.md — Evaluation](v1.0.md#evaluation).

### King Safety

1.1 adds a king safety penalty: a quadratic penalty for enemy pieces that attack the king zone.

#### King Zone

The king zone is the king's square plus all adjacent squares — up to 9 squares in the center, fewer near the edges:

```cpp
Bitboard king_zone = king_attack(king_sq) | square_bb(king_sq);
```

#### Attack Weights

Each enemy piece type has an attack weight:

| Piece | Weight |
|-------|--------|
| Pawn | 0 (handled implicitly by PST pawn shield) |
| Knight | 2 |
| Bishop | 2 |
| Rook | 3 |
| Queen | 5 |
| King | 0 |

Queens are counted **twice**: once when scanning diagonal pieces (as a bishop, weight 2) and once when scanning straight pieces (as a rook, weight 3). Total queen contribution: 5. This is intentional — a queen attacking the king zone is more dangerous than any single piece, and the double-count naturally models that without special-casing.

#### Penalty Formula

```cpp
Score penalty = total_weight * total_weight * SAFETY_SCALE;
return -(penalty * phase_mg) / 256;
```

The penalty is quadratic in the total attack weight. This means:

- One attacker barely matters (1² × 3 = 3 centipawns)
- Two attackers cause a moderate penalty (4² × 3 = 48 centipawns with weight 4)
- A coordinated attack with three or more pieces causes a severe penalty

`SAFETY_SCALE = 3` is a tuning constant. It can be increased to make the engine more aggressive about king attack, or decreased to reduce it.

#### Game Phase Scaling

The penalty is multiplied by `phase_mg / 256`:

- Full middlegame (phase_mg = 256): full penalty applied
- Full endgame (phase_mg = 0): no penalty (both kings should centralize)

This prevents the engine from avoiding king attacks in the endgame when king safety is largely irrelevant.

#### Symmetry

King safety is evaluated for both sides independently and subtracted:

```cpp
white_score += king_safety(board, WHITE, phase_mg);
black_score += king_safety(board, BLACK, phase_mg);
Score score = white_score - black_score;
```

If White's king is under attack, `white_score` is reduced. If Black's is, `black_score` is reduced. The difference reflects which king is safer.

## Transposition Table

### Storage and Probe

Unchanged from 1.0. See [v1.0.md — Transposition Table](v1.0.md#transposition-table).

### `probe_move()` Removed

The `probe_move()` convenience method was removed in 1.1. It was used in an earlier iteration of the bestmove resolution logic that relied on probing the TT after `negamax()` returned. That logic was replaced by `root_best_move_` (see [Stability Fixes](#stability-fixes)), making `probe_move()` dead code.

### TT Pollution Prevention

When `abort_search_` is set upon returning from a child node, the result is discarded without calling `TT.store()`:

```cpp
if (abort_search_) return 0;
```

This check appears both after `negamax()` returns and after `quiescence()` returns. Without it, a score of 0 from an incomplete search would be stored in the TT, and the next iteration would find a "deep" entry with a garbage score and potentially return an incorrect result.

## Time Management

### Soft/Hard Limit Model

1.0 used a single time budget per move. 1.1 replaces this with two limits:

- **Soft limit**: the normal target. After each completed iteration, if `elapsed >= soft_limit`, the search stops. The soft limit can be extended dynamically.
- **Hard limit**: the absolute ceiling. The search stops immediately (via `abort_search_`) when `elapsed >= hard_limit`, even mid-iteration.

The relationship between them is controlled by two constants:

```cpp
constexpr double SOFT_FACTOR = 0.6;  // soft_limit = base_time * 0.6
constexpr double HARD_FACTOR = 3.0;  // hard_limit = base_time * 3.0
```

A typical move on a 10-second clock with 30 moves remaining:
- Base time: `10000 / 30 ≈ 333ms`
- Soft limit: `333 * 0.6 ≈ 200ms`
- Hard limit: `333 * 3.0 = 999ms` (capped at 50% of remaining clock = 5000ms)

The engine normally finishes within the soft limit. The extra headroom up to the hard limit is reserved for dynamic extensions.

### Dynamic Extension

After each completed iteration, two conditions can trigger a soft limit extension:

1. **PV instability** — the best move changed from the previous iteration:
   ```cpp
   if (last_move != prev_move && last_move != MOVE_NONE)
       TM.extend_time(1.5);  // +50%
   ```
   The engine is reconsidering its top choice. Changing the PV late in the search often indicates the position is complex or a new threat was discovered.

2. **Score drop** — the evaluation fell by ≥ 30 centipawns from the previous iteration:
   ```cpp
   if ((prev_score - last_score) >= SCORE_DROP_THRESHOLD)
       TM.extend_time(1.25);  // +25%
   ```
   The engine found something bad — extra time helps find the best defensive response.

Both conditions can apply in the same iteration (multiplicative effect). The extended soft limit is always capped at the hard limit.

```cpp
void TimeManager::extend_time(double factor) {
    soft_limit_ms = int(soft_limit_ms * factor);
    soft_limit_ms = std::min(soft_limit_ms, hard_limit_ms);
}
```

### Supported Modes

Unchanged from 1.0: normal clock, `movetime`, `depth`, `infinite`. See [v1.0.md — Time Management](v1.0.md#time-management).

## Stability Fixes

These changes do not affect playing strength directly but eliminate crashes, illegal move outputs, and incorrect bestmove responses that were present in 1.0.

### Abort Flag (`abort_search_`)

**Problem**: in 1.0, the search called `TM.should_stop()` inside the recursion and returned immediately when time was up. This left `make_move()` calls unmatched — the board state was corrupted for the next search.

**Fix**: instead of returning directly, the search sets `abort_search_ = true` and every level of the call stack checks the flag at entry:

```cpp
Score Search::negamax(Board& board, Score alpha, Score beta, int depth, int ply) {
    if (abort_search_) return 0;
    // ...
    if ((stats_.nodes & 2047) == 0 && TM.should_stop()) {
        abort_search_ = true;
        return 0;
    }
    // ...
    board.make_move(m);
    Score score = -negamax(board, -beta, -alpha, depth - 1, ply + 1);
    board.unmake_move(m);  // Always executes — abort unwinds the stack cleanly
    if (abort_search_) return 0;
    // ...
}
```

The flag is reset to `false` at the start of each `go()` call.

### Root Best Move Tracking (`root_best_move_`)

**Problem**: in 1.0, the bestmove was retrieved by probing the TT after the search. With an always-replace TT that was 90%+ full, the root entry was frequently overwritten mid-search, causing `bestmove 0000` outputs that fastchess interpreted as a disconnect.

**Fix**: `root_best_move_` is a member variable updated directly inside `negamax()` whenever a new best move is found at `ply == 0`:

```cpp
if (score > best_score) {
    best_score = score;
    best_move  = m;
    if (ply == 0) root_best_move_ = m;
    // ...
}
```

The search **never does TT early returns at the root**. When a TT hit occurs at `ply == 0`, the search continues through the move loop anyway:

```cpp
if (tt_entry.depth >= depth && ply > 0) {  // Note: ply > 0 only
    Score tt_score = score_from_tt(tt_entry.score, ply);
    if (...) return tt_score;
}
```

This guarantees `root_best_move_` is always set from the actual move loop — never from a potentially stale or collided TT entry.

### `safe_move` Pre-Seed

**Problem**: if depth 1 is aborted before evaluating a single node (extreme time pressure, e.g. < 10ms on the clock), `root_best_move_` would be `MOVE_NONE` and the engine would output `bestmove 0000`.

**Fix**: before the iterative deepening loop starts, the first legal move in the position is found and stored in `safe_move`:

```cpp
Move safe_move = MOVE_NONE;
{
    MoveList seed_moves;
    generate_all_moves(board, seed_moves);
    for (int i = 0; i < seed_moves.count; i++) {
        if (board.is_legal(seed_moves.moves[i])) {
            safe_move = seed_moves.moves[i];
            break;
        }
    }
}
```

`safe_move` is updated to a better move as soon as any iteration completes. The final bestmove selection:

```cpp
result.best_move = (last_move != MOVE_NONE) ? last_move : safe_move;
```

This guarantees `bestmove` is always a legal move.

### `is_legal()` Piece Ownership Check

**Problem**: `is_legal()` in 1.0 worked by copying the board, calling `make_move()`, and checking if the king was attacked. If the from-square was empty (due to a stale TT entry from an earlier position), `make_move()` would "move nothing", the king would not be in check, and `is_legal()` would return `true` — approving a ghost move. This produced `Illegal PV move` warnings in fastchess during long games where the TT was full of entries from earlier positions.

**Fix**: `is_legal()` now checks piece ownership before the expensive board copy:

```cpp
bool Board::is_legal(Move m) const {
    Square from = from_sq(m);
    Piece  p    = piece_on[from];

    // Reject if from-square is empty or holds the opponent's piece
    if (p == NO_PIECE || color_of(p) != side_to_move)
        return false;

    Board copy = *this;
    copy.make_move(m);
    return !copy.is_attacked(copy.king_square(~copy.side_to_move),
                              copy.side_to_move);
}
```

This O(1) check eliminates the most common class of ghost moves from TT collisions at minimal cost.

## Known Limitations

The following limitations from 1.0 were addressed in 1.1:

- ~~**No killer moves**~~ — Fixed in 1.1.
- ~~**No king safety**~~ — Fixed in 1.1.
- ~~**Fixed time budget**~~ — Fixed in 1.1 with soft/hard limits and dynamic extension.

The following limitations remain and are planned for future versions:

- **No null move pruning**: the search explores every branch without skipping turns.
- **No history heuristic**: quiet move ordering beyond killers is still random.
- **No late move reductions (LMR)**: all moves are searched at full depth.
- **No SEE (Static Exchange Evaluation)**: the quiescence search cannot distinguish winning captures from losing ones.
- **Basic evaluation**: no pawn structure analysis, no mobility, no open file detection.
- **Always-replace TT**: a depth-based replacement strategy would retain more useful entries under pressure. Also limits PV chain extraction to a single move.
- **Single-threaded**: no support for Lazy SMP or any other form of parallelism.
- **No ponder**: the engine does not think on the opponent's time.
