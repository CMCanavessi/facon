## Overview

Facón 1.3 "Yunque" focuses on search selectivity (LMR, aspiration windows), positional evaluation (pawn structure), and intelligent time management. The combination produces +200 Elo over 1.2.

Unchanged from 1.2: board representation, magic bitboards, move generation, NMP, triangular PV array, PV repetition detection, king safety, killer heuristic, UCI threading, UCI option handling, heartbeat, currmove output.

## Pre-work Bug Fixes

### NMP depth floor

**Root cause**: the NMP recursive call passed `depth - 1 - NMP_REDUCTION` without a floor. At `depth == NMP_MIN_DEPTH` (3), this produced `3 - 1 - 3 = -1`. In 1.2 this silently entered quiescence via `depth <= 0`. With the corrected `depth == 0` condition (prerequisite for LMR), depth -1 falls through to full search and causes infinite recursion.

**Fix**: `std::max(0, depth - 1 - NMP_REDUCTION)`.

### `depth <= 0` changed to `depth == 0`

**Root cause**: LMR can produce reduced depths that are already 0. With `depth <= 0`, a negative depth from a buggy reduction would silently enter quiescence — masking the error. With `depth == 0`, negative depths cause a detectable hang. All reduction call sites now use `std::max(0, reduced_depth)`.

## Search — Late Move Reductions

### Concept

In a well-ordered move list, the first few moves are almost always best (TT move, good captures, killers). Later quiet moves are unlikely to raise alpha. LMR searches them at reduced depth first — if they fail low, no time is wasted. If they surprise by raising alpha, a full-depth re-search confirms the result.

### Implementation

```cpp
// In negamax(), after make_move(m):
bool do_lmr = (depth >= LMR_MIN_DEPTH)
           && (legal_count > LMR_MIN_MOVES)
           && !in_check
           && !is_capture
           && (move_type(m) != PROMOTION)
           && !is_killer;

if (do_lmr) {
    int reduction     = int(log(depth) * log(legal_count) / LMR_DIVISOR);
    reduction         = std::max(1, reduction);
    int reduced_depth = std::max(1, depth - 1 - reduction);

    // Zero-window at reduced depth
    score = -negamax(board, -alpha - 1, -alpha, reduced_depth, ply + 1);

    // Re-search at full depth if it didn't fail low
    if (score > alpha)
        score = -negamax(board, -beta, -alpha, depth - 1, ply + 1);
} else {
    score = -negamax(board, -beta, -alpha, depth - 1, ply + 1);
}
```

Constants: `LMR_MIN_DEPTH = 3`, `LMR_MIN_MOVES = 3`, `LMR_DIVISOR = 2.25`.

### Eligibility

LMR is skipped for moves that are likely to be important:

| Condition | Reason |
|-----------|--------|
| `is_capture` | Captures change material balance — must evaluate fully |
| `EN_PASSANT` | Classified as capture (included in `is_capture` check) |
| `PROMOTION` | Free piece — always important |
| `is_killer` | Already known to cause beta cutoffs at this ply |
| `in_check` | Evasions are critical — no reductions |

### Reduction formula

`reduction = log(depth) * log(move_number) / LMR_DIVISOR`

- Logarithmic in both depth and move number: deeper searches and later moves get larger reductions.
- Floored at 1 (always reduce by at least one ply).
- `reduced_depth` floored at 1 (never LMR directly into quiescence).
- `LMR_DIVISOR = 2.25`: larger value = less aggressive reductions.

### Depth floor rationale

`reduced_depth = std::max(1, depth - 1 - reduction)` ensures the reduced search always enters `negamax()` at depth >= 1, not quiescence. If LMR reduced directly to depth 0, all quiet moves beyond `LMR_MIN_MOVES` would effectively be pruned — quiescence only searches captures. Floor at 1 ensures at least one ply of full search before quiescence takes over.

## Search — History Heuristic

### Concept

Track which quiet moves have historically caused beta cutoffs. Use this information to order quiet moves: moves that have been good before are searched earlier and are less likely to be LMR-reduced.

### Implementation

```cpp
// search.h
static constexpr int HISTORY_MAX = 50000;  // Below ORDER_KILLER2 (80000)
int history_[2][64][64];                   // [color][from][to]
```

On beta cutoff by a quiet move:

```cpp
int bonus = depth * depth;
int& entry = history_[board.side_to_move][from_sq(m)][to_sq(m)];
entry = std::min(entry + bonus, HISTORY_MAX);
```

In `move_score()`, quiet moves return `history_[us][from][to]` instead of 0.

### Design decisions

- **`depth^2` bonus**: deeper cutoffs are more significant — a cutoff at depth 10 contributes 100 vs 9 at depth 3.
- **Cap at 50,000**: prevents history scores from exceeding `ORDER_KILLER2` (80,000), which would break the ordering tier hierarchy.
- **Reset each search**: entries from a previous position can mislead ordering. Cleared with `memset` at the start of `go()`.

## Search — Aspiration Windows

### Concept

Instead of searching each iteration with the full window `(-INFINITE, +INFINITE)`, use a narrow window centered on the previous iteration's score. Most positions are stable between iterations — the true score falls inside the window, and the narrower bounds cause more cutoffs (fewer nodes searched). When the score falls outside, widen the window and re-search.

### Implementation

```cpp
// In go(), for depth >= 4:
Score alpha_asp = last_score - ASP_WINDOW;
Score beta_asp  = last_score + ASP_WINDOW;
Score delta     = ASP_WINDOW;

while (true) {
    score = negamax(board, alpha_asp, beta_asp, depth, 0);
    if (abort_search_) break;

    if (score <= alpha_asp) {
        // Fail-low: widen alpha only, leave beta unchanged
        alpha_asp = std::max(score - delta, -SCORE_INFINITE);
        delta *= 2;
    } else if (score >= beta_asp) {
        // Fail-high: widen beta only, leave alpha unchanged
        beta_asp = std::min(score + delta, SCORE_INFINITE);
        delta *= 2;
    } else {
        break;  // Score fits inside the window
    }
}
```

Constants: `ASP_WINDOW = 50` (centipawns).

### Bypass conditions

| Condition | Reason |
|-----------|--------|
| `depth < 4` | Scores too unstable at shallow depths for windowing to help |
| `last_score == 0` | No prior score to center around (first iteration) |
| `is_mate_score(last_score)` | A +/-50cp window around a mate score wraps or sits outside valid range |

### Fail-low fix (post-gauntlet)

The original fail-low handler set `beta_asp = (alpha_asp + beta_asp) / 2`, squeezing the upper bound. When TT and LMR interactions caused the score to rebound on re-search, this artificial ceiling triggered a spurious fail-high, forcing a third search (yo-yo effect). Fix: on fail-low, widen only alpha. On fail-high, widen only beta. Standard implementation.

### Verbosity

Fail-low and fail-high events emit `info string AW:` lines with the window before/after, delta, and elapsed time. Same prefix style as TM messages.

## Evaluation — Pawn Structure

### Concept

Pawn structure is the most fundamental positional factor after material. Five terms capture the most important pawn relationships, all computed via bitboard operations (no per-square loops for the main detection).

### Terms

| Term | Value | Detection |
|------|-------|-----------|
| Isolated | -15cp per pawn | No friendly pawns on adjacent files |
| Doubled | -15cp per extra | More than one pawn on the same file |
| Backward | -12cp per pawn | Stop square attacked by enemy, no friendly support, not passed |
| Passed | +10/20/35/55/80cp by rank | No enemy pawn can block or attack the path to promotion |
| Connected | +8cp per pawn | Diagonally supported by a friendly pawn |

### Implementation

The function `pawn_structure()` returns a score from White's perspective.

**Isolated and doubled** are detected by file loop (0..7):

```cpp
for (int f = 0; f < 8; f++) {
    Bitboard pawns_on_file = our_pawns & file_bb(f);
    if (!pawns_on_file) continue;
    int count = popcount(pawns_on_file);
    if (count > 1)
        score += PAWN_DOUBLED * (count - 1);
    if (!(our_pawns & adjacent_files_bb(f)))
        score += PAWN_ISOLATED * count;
}
```

**Backward, passed, connected** are detected by individual pawn loop:

- **Passed**: build the front span and attack-forward span for this pawn. If no enemy pawn intersects either, it's passed. Bonus indexed by rank.
- **Backward**: stop square attacked by enemy pawn AND this pawn is not in the friendly attack span (no friend can push to support it). Not counted if already passed (would double-count).
- **Connected**: if any friendly pawn attacks this pawn's square (detected via `pawn_attacks_bb(~c, square_bb(sq)) & our_pawns`).

### Bitboard helpers

- `fill_north(b)` / `fill_south(b)`: flood-fill a bitboard through all ranks in one direction via repeated shifts (`b |= b << 8; b |= b << 16; b |= b << 32`).
- `file_bb(f)`: all squares on file `f`.
- `adjacent_files_bb(f)`: all squares on files adjacent to `f` (not including `f`).
- `pawn_attacks_bb(c, pawns)`: all squares attacked by a set of pawns of color `c`.

## Evaluation — Mopup Insufficient Material Guard

### Problem

K+B (330cp) and K+N (320cp) exceed `MOPUP_THRESHOLD` (300cp) and activated mopup corner-chasing in positions that are theoretical draws. The engine would refuse draws and chase the opponent's king indefinitely.

### Fix

At the top of `mopup_eval()`, check if the strong side has exactly two pieces (king + one piece). If that piece is a bishop or knight, return 0:

```cpp
Bitboard strong_pieces = board.all_pieces(strong_side);
int strong_count = popcount(strong_pieces);  // includes king
if (strong_count == 2) {
    if (board.piece_bb(strong_side, BISHOP) || board.piece_bb(strong_side, KNIGHT))
        return 0;
}
```

This correctly handles K+B vs K and K+N vs K. K+R vs K (500cp) and K+Q vs K (900cp) still activate mopup normally — these are winnable.

## Time Management

### Quadratic extension scaling

Extensions in `go()` are pre-scaled by `(depth^2 / EXTENSION_FULL_DEPTH^2)`:

```cpp
double depth_scale = std::min(1.0,
    double(depth) * double(depth) /
    double(EXTENSION_FULL_DEPTH) / double(EXTENSION_FULL_DEPTH));

// PV change: factor = 1.0 + 0.5 * depth_scale
// Score drop: factor = 1.0 + 0.25 * depth_scale
```

`EXTENSION_FULL_DEPTH = 18`. At depth 9, `depth_scale = 0.25` — a PV change extends by only x1.125 instead of x1.5. At depth 18+, the full factor applies.

**Rationale**: PV changes at depth 2-9 are normal search behavior (the engine hasn't settled yet), not genuine instability. Without scaling, these early changes consumed the extension budget before the engine reached its operating depth.

### `accumulated_ext_` cap removed

The old 2.0x cap (`MAX_EXTENSION_FACTOR`) tracked the product of all extension factors. Near-zero quadratic factors at low depths consumed the budget: `1.001 * 1.003 * 1.01 * ... > 2.0` long before any real extension at depth 14+ could fire. Removing the cap lets the soft limit be bounded only by the hard limit.

`accumulated_ext_` is still tracked for diagnostic output but no longer enforces a ceiling.

### Easy move reduction

`reduce_time(factor, reason)` mirrors `extend_time()`. Multiplies the soft limit by a factor < 1.0, floored at `MIN_TIME_MS` (10ms). Three triggers:

| Trigger | Factor | Condition |
|---------|--------|-----------|
| Mate found | x0.05 | `is_mate_score(last_score)` at depth > 12 |
| Forced move | x0.10 | `root_move_count_ == 1` at depth > 12 |
| Easy move | x0.40 | `stable_iters_ == 7` at depth > 12 |

**One-shot guards**: mate uses `mate_reduction_applied_` (without it, x0.05 fires every iteration → x0.05^N collapse). Easy move uses `stable_iters_ == 7` (fires exactly once at the threshold, not `>=`).

**Cancel on instability**: if a PV change or score drop triggers an extension after an easy-move reduction was applied, `cancel_easy_move(EASY_REDUCE_FACTOR)` restores the soft limit by dividing by the factor before applying the extension. This ensures extensions always act on the un-discounted soft limit.

### Emergency hard limit

When `depth >= EMERGENCY_DEPTH` (25) and the computed new soft would exceed the hard limit:

```cpp
// timeman.cpp — extend_time():
if (depth >= EMERGENCY_DEPTH && new_soft > hard_limit_ms) {
    int absolute_limit = raw_remaining_ms_ / 2;
    new_soft           = std::min(new_soft, absolute_limit);
    soft_limit_ms      = new_soft;
    hard_limit_ms      = new_soft;  // hard matches soft exactly
}
```

Both limits rise together on subsequent extensions, capped at 50% of the raw remaining clock (vs 33% in the normal path). This is triggered by real instability only — a stable position at depth 25+ would have already fired easy-move reduction, which would have stopped the search at the reduced soft limit.

`EMERGENCY_DEPTH` is a TM-internal constant, not exposed to search.

### Message format standardization

All TM and AW messages follow a consistent format:

```
info string TM: extend x1.5 (PV change) -- soft OLD -> NEW (hard H, headroom R) [T]
info string TM: reduce x0.4 (easy move) -- soft OLD -> NEW (hard H, headroom R) [T]
info string TM: cancel easy -- soft OLD -> NEW (hard H, headroom R) [T]
info string AW: fail-low depth D score Scp -- window [a, b] -> [a', b'] delta D [T]
```

Two-letter prefix, description with `--` separator, before/after values, elapsed timestamp in `h:mm:ss,ms` format.

## Infrastructure

### Centralized version system

`PROJECT_VERSION` and `FACON_CODENAME` in `CMakeLists.txt` are the single source of truth. CMake generates `version.h` via `configure_file`:

```cmake
set(FACON_CODENAME "Yunque")  # "dev" during development

if(FACON_CODENAME STREQUAL "dev")
    # binary: facon-1.3-dev, UCI: "Facon 1.3-dev"
else()
    # binary: facon-1.3, UCI: "Facon 1.3 - Yunque"
endif()
```

Binary name, startup banner, UCI `id name`, and Windows version resource all derive from these two variables.

### `perft` command

`perft N` counts leaf nodes to depth N. `perft divide N` gives per-move breakdown. Bulk-counting at depth 1 (count legal moves directly, skip make/unmake). Operates on a copy of the board — current position is not modified.

Verified: startpos depth 5 = 4,865,609 (matches published value).

### `bitboard.cpp` init message gated

"Initializing magic bitboards... done." now prints only when stderr is a terminal (`isatty()`). Suppressed when launched by GUI or fastchess.

## Bug Fixes

### Aspiration window fail-low (correctness)

The original fail-low handler set `beta_asp = (alpha_asp + beta_asp) / 2`. This squeezed the upper bound. When TT entries or LMR interactions caused the score to rebound on re-search, the narrowed beta ceiling triggered a spurious fail-high, forcing a third search. In unstable positions this could produce a yo-yo pattern across multiple re-searches.

**Fix**: on fail-low, widen alpha only and leave beta unchanged. On fail-high, widen beta only and leave alpha unchanged.

### Mate reduction one-shot (time management)

`is_mate_score(last_score)` is true on every iteration after a mate is first found. `reduce_time(0.05, "mate found")` was firing at depths 13, 14, 15..., causing `x0.05^N` collapse of the soft limit. At depth 15 (only 2 iterations after finding mate), the soft limit was already `0.05^2 = 0.25%` of original.

**Fix**: `mate_reduction_applied_` bool, set to true on first fire, checked before applying. Same pattern as `easy_move_applied_`.

### Race condition in `cmd_ucinewgame()` (threading)

If the search thread was still active when `ucinewgame` arrived, `TT.clear()` (`std::memset`) would race with `TT.probe()`/`TT.store()` in the search thread — undefined behavior.

**Fix**: `if (search_thread_.joinable()) { TM.stop = true; search_thread_.join(); }` before clearing the TT and resetting the board.

### Castling SAN check/mate (output)

`move_to_san()` returned immediately for castling moves (`return "O-O"` / `return "O-O-O"`) before reaching the check/checkmate detection block at the end of the function. Castling moves that deliver check or mate were missing the `+` or `#` suffix.

**Fix**: changed the castling branch from `return` to `san =`, so execution falls through to the check/mate detection block that runs for all move types.

### `seen[]` guard mismatch (PV display)

The PV repetition detection array was enlarged to `MAX_GAME_HISTORY + MAX_PLY + 2` (1154 slots) in 1.2, but the insertion guard still used `MAX_GAME_HISTORY + MAX_PLY` (1152), stopping 2 entries short. Updated the guard to match the array size.

## Known Limitations

- **`movestogo` ignored**: the engine always assumes 30 moves remaining. Tournaments using `movestogo` time controls may see suboptimal time allocation.
- **`TTEntry::depth` is `int8_t`**: max 127, but `MAX_PLY` is 128. Theoretical overflow at depth 128 (practically unreachable).
- **No check extension**: when in check, the search does not extend depth. LMR correctly skips reduction in check, but an active extension would improve tactical accuracy.
- **No SEE**: captures are ordered by MVV-LVA but not filtered by Static Exchange Evaluation. Losing captures (e.g. QxP when the queen is recaptured) are searched at full depth.
- **No quiet promotions in qsearch**: `generate_captures()` skips quiet promotions. A pawn push to the 8th rank (queen for free) is not found in quiescence.
- **`is_legal()` via board copy**: each legality check copies the entire `Board` struct (~700+ bytes). The make/unmake approach planned for 1.4 eliminates this.
- **`is_repetition()` detects twofold, not threefold**: returns true on the second occurrence. Standard for search purposes but worth noting.
- **LMR table not precalculated**: `log(depth) * log(move) / LMR_DIVISOR` is computed per-move at runtime. Precalculating into a lookup table is planned for 1.4.
